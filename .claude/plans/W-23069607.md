# W-23069607 — Migrate sf.org.login.access.token to services

## Context

- Cmd: `sf.org.login.access.token` ("SFDX: Authorize an Org using Session ID") — gated by WI 3 (W-23069601 orgOpen, W-23069593 gatherers/pickers landed).
- File: `packages/salesforcedx-vscode-org/src/commands/auth/orgLoginAccessToken.ts` — `SfCommandlet` + `OrgLoginAccessTokenExecutor extends LibraryCommandletExecutor`.
- Executor `run()`: `AuthInfo.create({ accessTokenOptions: { accessToken, instanceUrl } })` (NO `loginUrl`) → `handleAliasAndDefaultSettings({ alias, setDefault:true, setDefaultDevHub:false })` → `updateConfigAndStateAggregators()`; catch `Bad_OAuth_Token` → channel append + rethrow.
- CURRENT UX on any thrown error (incl. Bad_OAuth_Token), via LibraryCommandletExecutor catch (commandletExecutors.ts:172-181): `notificationService.showFailedExecution` → ERROR TOAST ("... failed to run") + `channelService.appendLine(e.message)` + show channel. Error toast already shown today. Friendly bad-oauth line appended by `run`'s own catch BEFORE rethrow; executor also toasts + appends raw `e.message`. Migration must preserve: friendly channel message + error toast.
- Gatherer: `AccessTokenParamsGatherer` — PromptService-migrated (W-23069593); `gatherAccessTokenParams()` prompts instanceUrl (validateUrl), alias (empty→`DEFAULT_ALIAS`), accessToken (`password:true`); cancel → `UserCancellationError`; returns `AccessTokenParams`; NOT exported (class-only).
- Target pattern: `orgOpenCommand` (orgOpen.ts) — `Effect.fn('name')`, `api = yield* (yield* ExtensionProviderService).getServicesApi`, `registerCommandWithLayer(AllServicesLayer)` in `activateEffect`.
- ConnectionService (services pkg, connectionService.ts:90): private `createWebAuthInfo` — `AuthInfo.create({ accessTokenOptions: { accessToken, loginUrl, instanceUrl } })` then FORKS `authInfo.save()` in background (Effect.fork, line 107) and does NOT call `handleAliasAndDefaultSettings`. Its fork pattern is NOT reusable for login: login must AWAIT save + alias/default before the command reports success. Plan: add `loginWithAccessToken` accessor that does create → `save()` (awaited, not forked) → `handleAliasAndDefaultSettings` (awaited). Exposes only `getConnection`/`invalidateCachedConnections`/`listAllAuthorizations` today.
- ConnectionService in services API type map (`index.ts:116`); consumed via `api.services.ConnectionService.*`. Existing typed errors (connectionService.ts): `FailedToCreateAuthInfoError` (create), `FailedToSaveAuthInfoError` (raised ONLY by `authInfo.save()`, line 112). Do NOT overload `FailedToSaveAuthInfoError` for the distinct alias/setDefault step — see Phase 1.
- `getBaseMessage` (errorHandlerService.ts:38) prefers inner `cause.message` over the error's own `message`. Tagged error needing friendly toast/channel text → construct WITHOUT `cause` (else raw `Bad_OAuth_Token` cause text surfaces). BadOAuthTokenError carries only localized `message`, no `cause`.
- `updateConfigAndStateAggregators` (orgUtil.ts) — keep; invalidates conn cache + refreshes aggregators.
- ChannelService: `appendToChannel`, `showChannel` — pattern from orgOpen.ts.
- No CLI involved — already CLI-free; migration = SfCommandlet/executor → Effect service command.

## Done when

- `sf.org.login.access.token` = `Effect.fn` cmd registered via `registerCommandWithLayer`.
- Prompts via existing gatherer Effect (PromptService). Auth via ConnectionService accessor (no direct `AuthInfo.create` in org cmd). Bad_OAuth_Token → friendly channel message + error toast preserved (re-fail through ErrorHandlerService, matching current `showFailedExecution` toast).
- Old `SfCommandlet`/`OrgLoginAccessTokenExecutor`/`orgLoginAccessToken` async export removed; `vscode.commands.registerCommand('sf.org.login.access.token', ...)` dropped from `registerCommands()`.
- Jest + new e2e green.

## Phases

### Phase 1 — ConnectionService.loginWithAccessToken accessor

- `connectionService.ts` — add THREE `export class` tagged errors (Schema.TaggedError) at module top, near the existing ones (lines 34-87):
  - `BadOAuthTokenError` — fields: `message: Schema.String` ONLY (NO `cause`, so `getBaseMessage` (errorHandlerService.ts:41) returns the error's own `message`, not the raw `Bad_OAuth_Token` cause). Services owns the friendly copy: add key (e.g. `bad_oauth_token_message`) to services `messages/i18n.ts` and construct `new BadOAuthTokenError({ message: nls.localize('bad_oauth_token_message') })` in the create catch (services has its own nls — `messages/index.ts`). This lets the org command re-fail the ORIGINAL `e` (a value inside the catchTag callback) without value-importing the class, so the index.ts export stays type-only and ErrorHandlerService toasts the friendly text. (If product wants the existing org-pkg wording, copy that string into the services message key.)
  - `FailedToHandleAliasSettingsError` — NEW, distinct from `FailedToSaveAuthInfoError`. fields: `message`, `cause` (optional Error). Raised by the `handleAliasAndDefaultSettings` step so callers can distinguish save failure from alias/default failure. Do NOT reuse `FailedToSaveAuthInfoError` (semantically bound to `authInfo.save()` at line 112).
  - (`FailedToCreateAuthInfoError`, `FailedToSaveAuthInfoError` reused for create + save steps.)
  - add accessor `loginWithAccessToken({ instanceUrl, accessToken, alias, setDefault })` (Effect.fn):
    - `AuthInfo.create({ accessTokenOptions: { accessToken, loginUrl: instanceUrl, instanceUrl } })` via `Effect.tryPromise`. BEHAVIOR NOTE: old org code passed `{ accessToken, instanceUrl }` (NO `loginUrl`); this adds `loginUrl: instanceUrl` to match `createWebAuthInfo`. @salesforce/core defaults `loginUrl` from `instanceUrl` when omitted, so this is parity, not a change — confirm against the @salesforce/core AuthInfo accessToken path before relying on it; if uncertain, keep the old `{ accessToken, instanceUrl }` shape to preserve exact behavior. catch maps to `BadOAuthTokenError` when `error.message.includes('Bad_OAuth_Token')`, else `FailedToCreateAuthInfoError`. (Old code only checked the message on the COMBINED create+handle flow; Bad_OAuth_Token originates from `AuthInfo.create`, so checking the create catch is correct.)
    - `authInfo.save()` AWAITED (NOT forked — login must persist before reporting success; the createWebAuthInfo fork at line 107 is for the read-path connection cache and is not appropriate here) via `Effect.tryPromise` catch → `FailedToSaveAuthInfoError`.
    - `authInfo.handleAliasAndDefaultSettings({ alias, setDefault, setDefaultDevHub: false })` AWAITED via `Effect.tryPromise` catch → `FailedToHandleAliasSettingsError`.
    - `Effect.withSpan('ConnectionService.loginWithAccessToken')`, annotate `authInfo.getFields()` (getFields excludes the raw token; confirm no token value leaks into span).
    - return `authInfo.getFields()` (authFields) for telemetry/caller use.
  - add to service return obj (`accessors: true` then exposes `ConnectionService.loginWithAccessToken`).
- services index.ts (index.ts:179-186): add `BadOAuthTokenError` AND `FailedToHandleAliasSettingsError` to the `export type { ... } from './core/connectionService'` block. REQUIRED — both appear in `loginWithAccessToken`'s Effect error channel; without the export the org pkg `.d.ts` hits TS4023. Type-only export is sufficient: `catchTag('BadOAuthTokenError', cb)` needs only the string-literal tag (carried in the inferred error type) and the org pkg re-fails the original `e` value from the callback, so NO value-construction in the org pkg → no value export needed. Mark each `/** @ExportTaggedError */` if knip flags them unused.
