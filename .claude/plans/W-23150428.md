## Mechanism (load-bearing ‚Äî addresses adversary findings)

- `draftPrPrompt(chosen, identity, fixerResult)` is a plain template string handed to `agent()`. There is NO function calling ‚Äî the agent cannot invoke `extractDiscussionUrls`. The ONLY mechanism is: run `extractDiscussionUrls(chosen.details)` in JS scope BEFORE the prompt is built, then string-interpolate its output into the template (same pattern as `chosen.details` at L827).
- Insertion: build `const reconstructedUrls = extractDiscussionUrls(chosen.details)` at top of `draftPrPrompt` body (right after the `pathsFor` destructure, L894). Interpolate the joined list into the step-4 `## Summary`/reference area ‚Äî see Phase 1 for exact prompt text. If `reconstructedUrls` is empty, interpolate nothing (no empty placeholder, no instruction).
- The reconstructed URLs are precomputed strings injected verbatim; the agent is told to copy them into the PR body, not to derive them.

## Regex / re-tick safety (load-bearing ‚Äî addresses adversary findings)

- PR snippet written at L924 is `<a href="PR_URL">#PR_NUMBER</a>` ‚Üí link TEXT is `#NNN` (a number), the path `discussions/\d+`|`issues/\d+` is only in the `href`, which is a populated real URL. `extractDiscussionUrls` matches only EMPTY-href anchors by link TEXT, so the workflow's own PR snippet can never produce a ghost URL on re-tick. Verify this with a fixture (see Verification).
- "Empty href" must cover all SF storage forms: `href=""`, `href=" "` / whitespace-only, HTML-entity-encoded `href=&quot;&quot;`, and absent `href` entirely.
- Populated-href anchors (real `href="https://..."`) are EXCLUDED even when link text matches `discussions/\d+` ‚Äî these are user-authored live links; the URL is already present so no reconstruction is needed and reconstructing would risk overwriting a correct host/path.
