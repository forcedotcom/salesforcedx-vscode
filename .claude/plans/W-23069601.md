# W-23069601 — Migrate sf.org.open to services (POC)

## Context

- POC gating all 4.x service migrations. Land + review first; fold learnings into 4.x.
- Simplest CLI command chosen as proof: `org open --url-only --json` → `openExternal`.
- Current: `packages/salesforcedx-vscode-org/src/commands/orgOpen.ts` — `SfCommandlet` + 2 `SfCommandletExecutor`s:
  - `OrgOpenExecutor` (non-container): runs `sf org:open` via CLI, CLI opens browser.
  - `OrgOpenContainerExecutor` (container): `org:open --url-only --json` via `CliCommandExecutor`, parses stdout w/ `OrgOpenContainerResultParser`, `vscode.env.openExternal`, appends msg to channel.
- Model to copy: `orgDeleteDefaultCommand` (same file's sibling `orgDelete.ts`) — `Effect.fn` cmd, `api.services.TerminalService.simpleExec`, `api.services.ChannelService.appendToChannel`, registered via `registerCommandWithLayer(AllServicesLayer)` in `index.ts`.
- Services available: `TerminalService.simpleExec({command, parse, timeout})` (desktop-only, fails web w/ `TerminalServiceError`); `ChannelService.appendToChannel`; `ProjectService.isSalesforceProject()`; `TargetOrgRef` (SubscriptionRef w/ `username`).
- Hard constraints carried over from old container path (`orgOpen.ts:57-60, 119`) — all three are load-bearing, not optional:
  - clean JSON stdout: container path set `env: { SF_JSON_TO_STDOUT: 'true' }`. `--json` alone is NOT confirmed to guarantee mixed-text-free stdout across sf CLI versions; orgDelete's simpleExec didn't need it (parse: identity, raw text). orgOpen DOES `JSON.parse`, so this env is required.
  - cwd / default-org resolution: container path passed `cwd: getRootWorkspacePath()`. `simpleExec` runs with no workspace cwd; `sf org open` resolves the default org from config relative to cwd → must pass `--target-org <username>` (orgDelete pattern, orgDelete.ts:170-172).
  - project precondition: old path gated on `sfProjectPreconditionChecker`, which shows the localized `predicates_no_salesforce_project_found_text` notification when no SF project. Must be reproduced as an Effect precondition.
- Registration: `index.ts` `activateEffect` already does `registerCommand('sf.org.delete.default', orgDeleteDefaultCommand)`; `ORG_OPEN_COMMAND = 'sf.org.open'` currently registered via plain `vscode.commands.registerCommand(ORG_OPEN_COMMAND, orgOpen)`.
- Channel name = displayName `Salesforce Org Management`.
- `openExternal`: no service wrapper; call `vscode.env.openExternal(URI.parse(url))` inside `Effect.sync`/`Effect.promise` (orgDelete calls `vscode` directly via `Effect.sync`).

## Done when

- `sf.org.open` runs as `Effect.fn` command registered via `registerCommandWithLayer`.
- Uses `TerminalService.simpleExec` for `org open --url-only --json`, opens url via `openExternal`, appends access message to channel.
- Old `SfCommandlet`/executor path + `OrgOpenContainerResultParser` removed (or parser retained only if still referenced).
- Jest + new e2e green.

## Phases

### Phase 0 — extend TerminalService for env (clean JSON stdout)

- `org open --json` parsing depends on `SF_JSON_TO_STDOUT=true` (old container path, orgOpen.ts:59). Resolve at the service seam, not via string-prefix hacks:
  - `childProcess.ts`: add `env?: Record<string, string>` to `ExecOptions` type.
  - `childProcess.ts` `exec`: merge into the node exec call — `promisify(exec)(command, { ...options, env: { ...process.env, ...options.env } })` (spread `process.env` so PATH etc. survive; node `exec` env defaults to inheriting parent only when env is omitted — passing a partial env would otherwise drop PATH). Keep `timeout`/`signal` threading unchanged.
  - `terminalService.ts` `simpleExec`: add optional `env?: Record<string, string>` param; thread into `childProcess.exec(command, { timeout, signal, env })`. Annotate span w/ presence of env keys (not values — no secret leakage).
  - jest (services pkg): assert `simpleExec` forwards `env` to the injected `ChildProcess.exec` mock; assert `process.env` keys still present in merged env.
- commit: `feat(services): TerminalService.simpleExec accepts env override - W-23069601`

### Phase 1 — orgOpenCommand Effect + registration

- New `orgOpenCommand = Effect.fn('orgOpenCommand')` in `orgOpen.ts`:
  - get api.
  - **precondition (load-bearing, not optional)**: `const isProject = yield* api.services.ProjectService.isSalesforceProject()`. If `!isProject`: show localized error + bail without running CLI. Mirror `sfProjectPreconditionChecker` (preconditionCheckers.ts:19-43): it shows `predicates_no_salesforce_project_found_text` and (on cause) `predicates_no_folder_opened_text`. Reproduce via a tagged error (e.g. `NoSalesforceProjectError`) surfaced through the command's error channel, OR an `Effect.sync(() => void vscode.window.showErrorMessage(nls.localize(...)))` + early return. Prefer the tagged-error path so ErrorHandlerService renders it consistently — confirm ErrorHandlerService maps the tag to `showErrorMessage` (don't double-notify). `isSalesforceProject` also sets the `sf:project_opened` context as a side effect (parity w/ old checker). Note: `nls.localize('predicates_*')` lives in effect-ext-utils messages, not org pkg — localize in org pkg's own nls or call the checker's message keys via the services api if exposed; resolve exact source while building.
  - **target-org (avoids cwd dependency)**: `const orgInfo = yield* SubscriptionRef.get(yield* api.services.TargetOrgRef())`; build `const targetOrgFlag = orgInfo.username ? \` --target-org ${orgInfo.username}\` : ''` (same as orgDelete.ts:172). If no username, the command still runs (sf resolves default) but log/annotate the gap.
  - **env (clean JSON)**: `terminalService.simpleExec({ command: \`sf org open --url-only --json${targetOrgFlag}\`, parse, env: { SF_JSON_TO_STDOUT: 'true' }, timeout })` using the Phase 0 env param.
  - `parse`: `JSON.parse(stdout)` → typed `{ result: { url, orgId, username } }`; Effect Schema decode preferred over raw assert (handles malformed stdout w/ a tagged decode error instead of an unhelpful SyntaxError). Map the error-shape (`{ status: 1, message }`) too, to preserve the old `OrgOpenErrorResult` channel-message behavior (orgOpen.ts:83-88).
  - on url: `Effect.sync(() => void vscode.env.openExternal(URI.parse(url)))`.
  - append `org_open_container_mode_message_text` (orgId/username/url) via `api.services.ChannelService.appendToChannel`; `channelService.showChannelOutput()` parity w/ container path.
- `index.ts`: drop `vscode.commands.registerCommand(ORG_OPEN_COMMAND, orgOpen)` from `registerCommands()`; add `yield* registerCommand(ORG_OPEN_COMMAND, orgOpenCommand)` in `activateEffect` next to `sf.org.delete.default`.
- Remove old `OrgOpenExecutor`, `OrgOpenContainerExecutor`, `getExecutor`, `orgOpen`, `isSFContainerMode`, `OrgOpenContainerResultParser` usage. Delete `parsers/orgOpenContainerResultParser.ts` + its barrel/`commands/index.ts` export if unused elsewhere.
- Drop now-unused imports (`CliCommandExecutor`, `SfCommandlet`, `ProgressNotification`, `sfProjectPreconditionChecker`, `EmptyParametersGatherer`, `workspaceUtils`, etc.); add `SubscriptionRef`, `Schema` (for decode), `nls` keys as needed.
- commit: `refactor(org): migrate sf.org.open to services via TerminalService - W-23069601`

## Resolved (were open questions)

- Project precondition — REQUIRED, reproduced as Effect precondition via `ProjectService.isSalesforceProject()` (Phase 1). Old path's notification preserved.
- `--json` sufficiency — NOT relied upon; `SF_JSON_TO_STDOUT=true` passed explicitly via the new `simpleExec` `env` param (Phase 0), matching the old container path.
- cwd — replaced with `--target-org <username>` from `TargetOrgRef` (Phase 1), so no workspace cwd needed.
