## Context

- Target: `packages/salesforcedx-vscode-org/src/commands/orgCreate.ts` — current `SfCommandlet` (`sfProjectPreconditionChecker` + `CompositeParametersGatherer(FileSelector, AliasGatherer)` + `OrgCreateExecutor`) running `org:create:scratch` via `CliCommandExecutor`.
- Build (orgCreate.ts:53): `org:create:scratch --definition-file <rel> --alias <a> --duration-days <d> --set-default --json`. `--set-default` makes new org default; CLI cmd description = `org_create_default_scratch_org_text`.

- Precondition: `checkDevHubConfigured()` (devUsernameChecker.ts; Effect-based, returns bool, shows "no dev hub" warning) + `sfProjectPreconditionChecker`.
- Reference (POC) `orgOpen.ts`: `Effect.fn('orgOpenCommand')`, `api = yield* (yield* ExtensionProviderService).getServicesApi`, `ProjectService.getSfProject()` precondition (sets `sf:project_opened`, typed err), `TerminalService.simpleExec({command, parse, timeout?})`, Schema tagged-union decode + `Match.type<T>().pipe(Match.tag, Match.exhaustive)`, `ChannelService.appendToChannel`+`showChannel`.

- `updateConfigAndStateAggregators` (orgUtil.ts:115) — refresh after success (success path only, parity orgCreate.ts:89).
- Parser `parsers/orgCreateResultParser.ts` — `OrgCreateResultParser`/`OrgCreateErrorResult`. Replace w/ Schema tagged union; delete file + its export iff unreferenced elsewhere (grep first).
- nls keys (i18n.ts): `org_create_default_scratch_org_text`, `org_create_result_parsing_error`, `error_no_scratch_def`, `error_invalid_org_alias`, `error_invalid_expiration_days`, `parameter_gatherer_enter_alias_name`, `parameter_gatherer_enter_scratch_org_def_files`, `parameter_gatherer_enter_scratch_org_expiration_days`. Add progress-title key (e.g. `org_create_progress`).
- E2E: `orgPicker.desktop.spec.ts` (singular) already drives full create flow (file pick/alias/days). Its success assertion (line 128) waits on the `CREATE_SCRATCH_ORG_RAN` toast (`SFDX: Create a Default Scratch Org... successfully ran`). That toast is emitted ONLY by `SfCommandletExecutor` via `notificationService.reportCommandExecutionStatus`→`showSuccessfulExecution` (notificationService.ts:79,101) — the OLD path. The migrated Effect command does NOT go through that executor, so it will NEVER emit this toast → existing test would time out (600s). DECISION: do NOT re-add the toast. Migrated org cmds signal success via channel text (orgOpen.ts:106 `channel.appendToChannel`; orgOpen e2e asserts via `waitForOutputChannelText`, orgOpen.desktop.spec.ts:52, NOT a toast). Phase 3 MUST replace the line-128 toast assertion in the existing success test with a deterministic channel-text + status-bar assertion (mirror orgOpen). WI = "extend orgPicker e2e with a cancel case." `orgPickers.desktop.spec.ts` (plural, WI 2) = display/delete/logout pickers — unrelated.
- `checkDevHubConfigured` (devUsernameChecker.ts:19-28) is NOT Effect-callable: it internally calls `getOrgRuntime().runPromise(...)`. Wrapping it in `Effect.promise` nests a `runPromise` inside an Effect — the forbidden anti-pattern (effect-best-practices SKILL.md "FORBIDDEN — runSync/runPromise inside services"). The migrated devhub gate must yield `api.services.ConfigService.getTargetDevHub()` directly (returns `string | undefined`, configService.ts:79) — NOT call `checkDevHubConfigured`.
