# W-23069610 — Migrate sf.org.logout.all to services

## Context

- WI 4.6; gated by WI 3 (now landed on develop: f4bf62251 migrated pickers + auth gatherers).
- Target: `sf.org.logout.all`. `sf.org.logout.default` OUT of scope (shares file/helpers).
- Current `orgLogoutAll` (`packages/salesforcedx-vscode-org/src/commands/auth/orgLogout.ts`):
  - `SfCommandlet(sfProjectPreconditionChecker, SelectOrgsForLogout, OrgLogoutSelected)`.
  - `OrgLogoutSelected extends LibraryCommandletExecutor`; per-org loop: `AuthRemover.create()` → `removeAuth` → `removeOrgAliases` → `checkIsCurrentTargetOrg`/`doUnsetTargetOrg` → `checkIsCurrentTargetDevHub`/`doUnsetTargetDevHub`.
- `node_modules/@salesforce/core/lib/org/authRemover.js` `removeAuth` (L50-54) calls `unsetConfigValues` + `unsetAliases`. `unsetConfigValues` (L132) unsets ALL global+local config keys matching username/aliases → covers target-org AND target-dev-hub. Extension's `removeOrgAliases` / `doUnsetTargetOrg` / `doUnsetTargetDevHub` / `isCurrentTarget*` checks + `getAliasesForUsername` all redundant.
- Pattern to mirror: orgOpen POC (`commands/orgOpen.ts` + `index.ts` `registerCommandWithLayer(AllServicesLayer)`, next to `orgDeleteDefaultCommand`) — d4c8eac8e.
- `SelectOrgsForLogout` gatherer already Effect (multi-pick + confirm modal, throws `UserCancellationError`). Fold its body into the command; drop `SfCommandlet`/`ParametersGatherer` wrapper.
- `AuthRemover` from `@salesforce/core`: no AuthService in services pkg; wrap in local `Effect.fn` with `Effect.tryPromise` (rejects → typed `OrgLogoutError`, never a defect).
- New tagged error `OrgLogoutError` (`Schema.TaggedError`, `message` + `username` + optional `cause`) — `removeAuth`'s `catch` maps the core rejection to it so ErrorHandlerService renders a real message instead of swallowing it as a defect.
- `updateConfigAndStateAggregators` (orgUtil.ts L116-127) is already partly Effect: it does `configAggregatorProvider.reloadConfigAggregators()` + `StateAggregator.clearInstanceAsync()` then runs the existing `refreshConnection` Effect (orgUtil.ts L105-113, which invalidates ConfigService/ConnectionService via the services api). The command must NOT bounce back through the async `updateConfigAndStateAggregators` wrapper; compose the Effect directly (see Phase 1).
- `sfProjectPreconditionChecker` → replace with `ProjectService.getSfProject()` (orgOpen precondition pattern), keeps no-project error rendered by ErrorHandlerService.
- Default-path redundancy: out of scope for this WI.
  - `OrgLogoutDefault.run` (orgLogout.ts L122-127) still calls `checkIsCurrentTargetOrg` + `removeOrgAliases` + `doUnsetTargetOrg` after `removeAuth` — duplicates `unsetConfigValues`+`unsetAliases` (authRemover.js L50-54).
  - Phase 1 adds TODO comment flagging duplication; removal deferred to follow-up WI.
- e2e reality (drives Phase 3 design):
  - `fullyParallel: true` (createDesktopConfig.ts L27) + shared on-disk auth.
  - Sibling specs (`orgOpen.desktop`, `orgDeleteCommandVisibility.desktop`) share `MINIMAL_ORG_ALIAS` on-disk auth concurrently.
  - `createMinimalOrg` hard-pinned to `MINIMAL_ORG_ALIAS` + `tryUseExistingOrg` fast path (minimalScratchOrgSetup.ts L14-24) — cannot mint a second distinct org.
  - Real-logout needs a dedicated throwaway org — never the shared fixture org — but it must be reachable in CI.
  - CI gating constraint (drives Phase 3 org creation): `requireOrgInCI` (shared.ts L60-65) THROWS whenever `process.env.CI` is set, regardless of whether the org exists. It is only safe AFTER a `tryUseExistingOrg` miss for an alias that CI pre-seeds. orgE2E.yml only pre-seeds `MINIMAL_ORG_ALIAS` (env L50, create step L115-120). A `logoutThrowaway-${Date.now()}`/uuid alias is never pre-seeded → `requireOrgInCI` would throw before any create code runs → every CI run fails. So a timestamped+`requireOrgInCI` design is unusable. Two viable options below; plan picks (a).

## Phases

### Phase 1 — migrate orgLogoutAll to Effect command

Files: `src/commands/auth/orgLogout.ts`, `src/index.ts`, `src/commands/index.ts`, `src/constants.ts` (add `ORG_LOGOUT_ALL_COMMAND`).

- New `orgLogoutAllCommand = Effect.fn('orgLogoutAllCommand')`:
  - precondition `api.services.ProjectService.getSfProject()`.
  - inline picker+confirm (move `gather` body from `selectOrgsForLogout.ts`): build items via `getFreshAuthorizations` + `buildOrgQuickPickItems`, multi-pick, `considerEmptySelectionAsCancellation`, scratch/sandbox confirm via `PromptService.confirmOrThrow`.
  - `yield* Effect.forEach(usernames, removeAuth, { discard: true })` where
    ```ts
    const removeAuth = Effect.fn('orgLogoutAllCommand.removeAuth')(function* (username: string) {
      yield* Effect.annotateCurrentSpan('username', username);
      yield* Effect.tryPromise({
        try: () => AuthRemover.create().then(r => r.removeAuth(username)),
        catch: cause => new OrgLogoutError({ username, message: `Failed to log out of ${username}`, cause: String(cause) })
      });
    });
    ```
    Use `Effect.tryPromise` (NOT `Effect.promise` — that converts a core rejection into an unhandled defect; `tryPromise` produces a typed `OrgLogoutError` that ErrorHandlerService can render). Cite effect-best-practices: "throw / generic Error / untyped Promise.reject → Schema.TaggedError + Effect.fail".
  - Do NOT call any of `removeOrgAliases` / `doUnsetTargetOrg` / `doUnsetTargetDevHub` / `checkIsCurrentTargetOrg` / `checkIsCurrentTargetDevHub` / `getAliasesForUsername` from the new all-path command — `removeAuth` already covers them (authRemover.js L50-54). DELETE only the helpers the default path does NOT use (`doUnsetTargetDevHub`, `checkIsCurrentTargetDevHub`, `getAliasesForUsername` — grep-confirm zero default-path refs). KEEP `removeOrgAliases` / `doUnsetTargetOrg` / `checkIsCurrentTargetOrg` because `OrgLogoutDefault` still calls them (their default-path redundancy is the deferred follow-up, see TODO below).
  - final config/state refresh: compose as Effect, do NOT bounce through the async `updateConfigAndStateAggregators` wrapper.
    - Reuse the existing `refreshConnection` Effect (orgUtil.ts L105-113) directly — `yield*` it (it already invalidates ConfigService + ConnectionService via the services api). It is currently `const` (not exported); export it (or extract a shared exported Effect) so the command yields it instead of re-entering an async boundary.
    - The two non-Effect halves (`ConfigAggregatorProvider.reloadConfigAggregators()` + `StateAggregator.clearInstanceAsync()`) wrap in `Effect.tryPromise` (each can reject) mapping to a tagged error, OR call them inside an `Effect.fn` that `tryPromise`s the combined `updateConfigAndStateAggregators` body MINUS the `refreshConnection` runPromise. Net: one Effect graph, no nested `getOrgRuntime().runPromise`, no `Effect.promise` swallowing rejections.
  - errors: catch ONLY `UserCancellationError` via `Effect.catchTag('UserCancellationError', () => Effect.void)` (cancel = intentional no-op). Every other error — `OrgLogoutError`, project precondition, config-refresh failure — propagates untouched to `ErrorHandlerService` (the command wrapper / `registerCommandWithLayer`). NO `catchAll`, NO swallow-to-be-safe. Cite effect-best-practices: "catch sparingly; no catchAll". Confirm the orgOpen/orgDelete command wrapper routes uncaught failures to ErrorHandlerService (it does — mirror it; do not add a local catch).
- `index.ts`: drop `orgLogoutAll` import + `registerCommand('sf.org.logout.all', orgLogoutAll)`; add `registerCommand(ORG_LOGOUT_ALL_COMMAND, orgLogoutAllCommand)` in `activateEffect` beside orgOpen.
- Remove `OrgLogoutSelected`, `orgLogoutAll`, `SelectOrgsForLogout` usage. KEEP `SimpleGatherer` (default path L145 uses it). Delete `src/parameterGatherers/selectOrgsForLogout.ts` (only consumer = all path).
- `commands/index.ts`: drop `orgLogoutAll` export.
- Keep `orgLogoutDefault` + its helpers (`OrgLogoutDefault`, `resolveTargetOrg`, scratch gatherer) untouched; verify which deleted helpers it shared (`removeOrgAliases`, `doUnsetTargetOrg`, `checkIsCurrentTargetOrg` used by default too) — KEEP those, only remove all-path-exclusive ones (`doUnsetTargetDevHub`, `checkIsCurrentTargetDevHub`, `getAliasesForUsername` if unused by default).
- TODO comment in `OrgLogoutDefault.run` (orgLogout.ts ~L122-127) above the `checkIsCurrentTargetOrg`/`removeOrgAliases`/`doUnsetTargetOrg` block: note these duplicate `removeAuth`'s `unsetConfigValues`+`unsetAliases` and should be removed when the default path migrates (cite follow-up WI). Prevents the silent duplication accumulating / a false "removeAuth covers everything" invariant.

### Phase 3 — e2e: real removeAuth removes the org (REQUIRED before merge)

Files: `test/playwright/specs/orgPickers.desktop.spec.ts`; new throwaway-org helper `createThrowawayOrg(alias)` in `packages/playwright-vscode-ext/src/orgs/`; `.github/workflows/orgE2E.yml` (pre-seed the throwaway alias).

— do NOT reuse `MINIMAL_ORG_ALIAS`: shared concurrently with siblings, cancel steps depend on it staying authed, no happy-path coverage today.

CI org-availability — chosen approach (a), fixed pre-seeded alias (matches the nonTrackingOrg pattern in apexTestingE2E.yml L112-118/135-138):
- Use a FIXED throwaway alias `THROWAWAY_ORG_ALIAS = 'logoutThrowawayOrg'` (constant in `packages/playwright-vscode-ext/src/orgs/`, mirroring `MINIMAL_ORG_ALIAS`). NOT timestamped/uuid — a non-pre-seeded alias fails `requireOrgInCI`.
- orgE2E.yml: add a `create scratch org` step (next to the existing `MINIMAL_ORG_ALIAS` create, L115-120) that creates `logoutThrowawayOrg` in `minimal-project`; add `THROWAWAY_ORG_ALIAS: logoutThrowawayOrg` to the job `env` (alongside `MINIMAL_ORG_ALIAS` at L50) and to the run step env. This makes `tryUseExistingOrg(THROWAWAY_ORG_ALIAS)` hit in CI so `requireOrgInCI` is never reached. The throwaway org being logged out by the test is fine — it is single-purpose and re-seeded each run.
- `createThrowawayOrg(alias)`: mirror `createMinimalOrg` (tryUseExistingOrg fast path → `requireOrgInCI` only on miss → `sf org create scratch`). Same gate semantics, now valid because the alias IS pre-seeded.
- Rationale for (a) over inline create: inline `execAsync` create (option b — skip tryUseExistingOrg+requireOrgInCI, wrap in try/catch for devhub availability) also works but duplicates the create logic and hides CI devhub failures behind a catch; (a) reuses the established fixture pattern and fails fast on a missing devhub.
- KEEP the two existing cancel-path LOGOUT steps unchanged (they must keep `MINIMAL_ORG_ALIAS` alive).
- Add a FINAL `test.step` 'LOGOUT real: confirm removes auth' (after both cancel steps): run `sf.org.logout.all`, `expectOrgPickerListsOrg(THROWAWAY_ORG_ALIAS)`, `toggleMultiPickRow(THROWAWAY_ORG_ALIAS)`, Enter, then CONFIRM the modal (Enter / click confirm) — NOT Escape. Assert removal durably:
  - primary (durable, not racy): poll `sf org list --json` (jq/parse) until the throwaway username/alias is absent, or `sf org display -o <alias> --json` errors — per playwright-e2e "prefer durable on-disk/CLI signals over UI toasts".
  - secondary (UI): re-invoke `sf.org.logout.all`, assert the picker no longer lists the throwaway alias (negate `expectOrgPickerListsOrg` via `toBeHidden`/`not.toBeVisible`), then Escape.
- Cleanup: the test logs the throwaway org out (not deletes it). The scratch org itself is reaped by the orgE2E.yml cleanup step (mirror the `sf org delete scratch -o "$MINIMAL_ORG_ALIAS"` at L179 — add a `logoutThrowawayOrg` delete beside it, best-effort `|| true`). Locally (non-CI) `createThrowawayOrg` re-creates on next run via the tryUseExistingOrg miss; no per-test `afterAll` delete needed and none should log out `MINIMAL_ORG_ALIAS`.
- Apply playwright-e2e skill conventions (durable signals, no clipboard, background-run when executing).
