# W-23069610 — Migrate sf.org.logout.all to services

## Context

- WI 4.6; gated by WI 3 (now landed on develop: f4bf62251 migrated pickers + auth gatherers).
- Target: `sf.org.logout.all`. `sf.org.logout.default` OUT of scope (shares file/helpers).
- Current `orgLogoutAll` (`packages/salesforcedx-vscode-org/src/commands/auth/orgLogout.ts`):
  - `SfCommandlet(sfProjectPreconditionChecker, SelectOrgsForLogout, OrgLogoutSelected)`.
  - `OrgLogoutSelected extends LibraryCommandletExecutor`; per-org loop: `AuthRemover.create()` → `removeAuth` → `removeOrgAliases` → `checkIsCurrentTargetOrg`/`doUnsetTargetOrg` → `checkIsCurrentTargetDevHub`/`doUnsetTargetDevHub`.
- Redundancy proven: `node_modules/@salesforce/core/lib/org/authRemover.js` `removeAuth` (L50-54) calls `unsetConfigValues` + `unsetAliases`. `unsetConfigValues` (L132) unsets ALL global+local config keys matching username/aliases → covers target-org AND target-dev-hub. So extension's `removeOrgAliases` / `doUnsetTargetOrg` / `doUnsetTargetDevHub` / the `isCurrentTarget*` checks + `getAliasesForUsername` are all redundant.
- Pattern to mirror: orgOpen POC (`commands/orgOpen.ts` + `index.ts` `registerCommandWithLayer(AllServicesLayer)`, next to `orgDeleteDefaultCommand`) — d4c8eac8e.
- `SelectOrgsForLogout` gatherer already Effect (multi-pick + confirm modal, throws `UserCancellationError`). Fold its body into the command; drop `SfCommandlet`/`ParametersGatherer` wrapper.
- `AuthRemover` from `@salesforce/core`: no AuthService in services pkg; wrap in local `Effect.fn` (thin core call, no new service warranted).
- `sfProjectPreconditionChecker` → replace with `ProjectService.getSfProject()` (orgOpen precondition pattern), keeps no-project error rendered by ErrorHandlerService.
- Default-path redundancy is REAL but OUT OF SCOPE: `OrgLogoutDefault.run` (orgLogout.ts L122-127) still calls `checkIsCurrentTargetOrg` + `removeOrgAliases` + `doUnsetTargetOrg` AFTER `removeAuth`, duplicating `removeAuth`'s `unsetConfigValues`+`unsetAliases` (authRemover.js L50-54). This WI does NOT touch the default path's logic — only adds a TODO comment there (Phase 1) flagging the duplication for a follow-up WI. Do NOT claim removeAuth makes the kept helpers globally redundant: it makes them redundant *for the all-path only*; the default path keeps calling them (correctly cleaning the default-org pointer is harmless but duplicated work).
- e2e reality (drives Phase 3 design): the spec is one test reusing the SINGLE fixture org `MINIMAL_ORG_ALIAS` across DISPLAY/DELETE/LOGOUT steps; `fullyParallel: true` (createDesktopConfig.ts L27) + global CLI auth on disk means sibling specs (`orgOpen.desktop`, `orgDeleteCommandVisibility.desktop`) share that same on-disk auth concurrently. Logging out `MINIMAL_ORG_ALIAS` would destroy auth for every concurrent step/spec. `createMinimalOrg` is hard-pinned to `MINIMAL_ORG_ALIAS` + a `tryUseExistingOrg` fast path (minimalScratchOrgSetup.ts L14-24) — it CANNOT mint a second distinct org. Real-logout therefore needs a dedicated, uniquely-aliased throwaway scratch org created in-spec, logged out last, asserted absent — never the shared fixture org.

### Phase 3 — e2e: real removeAuth removes the org (REQUIRED before merge)

Files: `test/playwright/specs/orgPickers.desktop.spec.ts`; helper for a throwaway org (new `createThrowawayOrg(alias)` in `packages/playwright-vscode-ext/src/orgs/`, or inline `sf org create scratch -a <unique> --json` via `execAsync`).

Why this shape (do NOT merge real-logout into the existing LOGOUT step): the existing LOGOUT step (L96-105) and LOGOUT-cancel step (L107-113) BOTH depend on `MINIMAL_ORG_ALIAS` staying authed, and that org is shared concurrently with sibling specs (see Context). The current LOGOUT steps only cover CANCEL paths — happy-path removeAuth has zero coverage. Adding real-logout against the fixture org would destroy it mid-run.

- KEEP the two existing cancel-path LOGOUT steps unchanged (they must keep `MINIMAL_ORG_ALIAS` alive).
- Mint a dedicated throwaway scratch org with a UNIQUE alias (e.g. `logoutThrowaway-${Date.now()}` or a uuid) — distinct from `MINIMAL_ORG_ALIAS`, never the alias-pinned `createMinimalOrg`. Create it in a `test.step` (or `beforeAll`) before the real-logout step; org is authed by creation (no auth-seed needed). Gate on CI-org availability the same way `createMinimalOrg` does (`requireOrgInCI`) so it fails fast rather than silently skipping.
- Add a FINAL `test.step` 'LOGOUT real: confirm removes auth' (after both cancel steps): run `sf.org.logout.all`, `expectOrgPickerListsOrg(throwawayAlias)`, `toggleMultiPickRow(throwawayAlias)`, Enter, then CONFIRM the modal (Enter / click confirm) — NOT Escape. Assert removal durably:
  - primary (durable, not racy): poll `sf org list --json` (jq/parse) until the throwaway username/alias is absent, or `sf org display -o <alias> --json` errors — per playwright-e2e "prefer durable on-disk/CLI signals over UI toasts".
  - secondary (UI): re-invoke `sf.org.logout.all`, assert the picker no longer lists the throwaway alias (negate `expectOrgPickerListsOrg` via `toBeHidden`/`not.toBeVisible`), then Escape.
- Cleanup: throwaway org is logged out by the test itself; add an `afterAll`/`finally` `sf org delete scratch -p -o <alias>` (best-effort, ignore failure) so a confirm-modal regression that skips logout does not leak scratch orgs.
- Apply playwright-e2e skill conventions (durable signals, no clipboard, background-run when executing).

Commit: `test(org): e2e real logout removes auth via throwaway org - W-23069610`
