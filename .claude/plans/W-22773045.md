# W-22773045 — Remove unused i18n keys from salesforcedx-vscode-core

## Context

- `packages/salesforcedx-vscode-core/src/messages/i18n.ts` carries dead keys post-migration of commands to other extensions (apex-log, metadata, lwc, lightning, apex-testing, apex-replay-debugger).
- Lint rule `local/no-unused-i18n-messages` default `dynamicKeyPatterns` = `['^[A-Z][a-zA-Z0-9]*$']` (`packages/eslint-local-rules/src/noUnusedI18nMessages.ts:15`) — masks all metadata-type keys (`ApexClass`, `AccessControlPolicy`, ...). No core src ref to those types except `src/messages/i18n.ts`.
- `metadata_types_map_scraped.json` consumed only by `scripts/xsd/*` (XSD gen) — not by core src/test.
- Four files to keep in sync: `i18n.ts`, `i18n.ja.ts`, `package.nls.json`, `package.nls.ja.json`.

### Cross-package coupling risk (adversary:critical)

External extensions own the migrated commands and define their own `package.nls.json` entries (e.g., `salesforcedx-vscode-apex-log/package.nls.json:5` defines `apex_generate_class_text`; metadata, lwc, lightning, apex-testing, apex-replay-debugger likewise). Their playwright specs import `packageNls` from `../../../package.nls.json` — i.e., their *own* package, not core. Removing keys from core's `package.nls.json` does NOT break those specs.

The single broken edge: `packages/salesforcedx-vscode-core/test/playwright/setupHelpers.ts:36` reads `packageNls.apex_generate_class_text` from core's `package.nls.json`, but `apex_generate_class_text` is owned by `salesforcedx-vscode-apex-log` (`package.json:418`). Core's `package.json` no longer has `%apex_generate_class_text%`. This helper is wrongly coupled to core's nls and must be repointed to apex-log's nls (or removed) before the core key is deleted.

Audit must classify each candidate key by: (a) referenced in core's own `package.json` via `%key%` → keep; (b) referenced in core src/test → keep; (c) referenced only in another extension's spec via core's `package.nls.json` → repoint that spec, then delete; (d) zero refs anywhere → delete.

**Steps**

Prefer the literal — playwright e2e already runs against the live command palette, the string is the contract.

### Phase 2: audit + remove dead keys

**Files**
- `packages/salesforcedx-vscode-core/src/messages/i18n.ts`
- `packages/salesforcedx-vscode-core/src/messages/i18n.ja.ts`
- `packages/salesforcedx-vscode-core/package.nls.json`
- `packages/salesforcedx-vscode-core/package.nls.ja.json`

**Audit script** (one-shot node, not committed) — emits explicit per-key disposition:

For each key in core's `i18n.ts`:
1. Search core `src/` and `test/` for: `nls.localize('key'`, `coerceMessageKey('key'`, `messages.key`, `messages['key']`, bare quoted `'key'`. Exclude `messages/i18n*.ts`.
2. Search dynamic patterns: `Object.keys(messages)`, `messages[`<var>`]`. Document any hit; those keys cannot be safely removed without code change.

For each key in core's `package.nls.json`:
1. Search core's `package.json` for `%key%`.
2. Search **all other extensions' `test/`** trees (`packages/*/test/`) for `packageNls.key` where the importing file's relative `package.nls.json` resolves back to core. (Today: `core/test/playwright/setupHelpers.ts` only — handled in Phase 1. After Phase 1, no external test imports core's nls.)

**Disposition output** — script must print, e.g.:
```
KEEP (package.json %ref%): config_list_text, ...
KEEP (src ref): <list>
KEEP (test ref): <list>
REMOVE (zero refs): apex_generate_class_text, deploy_this_source_text, project_deploy_start_default_org_text, lightning_generate_lwc_text, project_generate_manifest, ...
REMOVE (metadata-type, no dynamic lookup): AccessControlPolicy, ApexClass, ..., XOrgHub  (count: ~310)
DYNAMIC (manual review): <list or empty>
```

**Steps**
- run audit; abort and surface for human review if `DYNAMIC` is non-empty.
- delete REMOVE keys from all four files. Maintain key-set parity: `i18n.ts` ⟷ `i18n.ja.ts`; `package.nls.json` ⟷ `package.nls.ja.json` (Japanese files mirror the English key set, values translated).
- commit message must enumerate categories and counts (e.g., `removes 5 migrated-command keys + ~310 metadata-type labels; see audit output in PR body`).

**Commit**: `chore(core): remove unused i18n keys`

### Phase 3: tighten lint rule for core (mandatory)

Mandatory because Phase 2 deletes ~310 metadata-type keys; without this, the next contributor adding `ApexClass` (or any `^[A-Z][a-zA-Z0-9]*$` key) hits zero lint feedback under the global default and the dead-key class regrows.

**Files**
- `eslint.config.mjs`

**Steps**
- add a per-file override block keyed on `packages/salesforcedx-vscode-core/src/messages/i18n.ts` (and `.ja.ts`) setting:
  - `'local/no-unused-i18n-messages': ['error', { dynamicKeyPatterns: [] }]`
- if Phase 2 audit surfaces legitimate dynamic refs (DYNAMIC non-empty), encode them as explicit `allowList` entries in the same override rather than a regex.
- re-run lint to confirm no false positives.

**Commit**: `chore(core): drop default dynamic-key pattern for i18n lint`
