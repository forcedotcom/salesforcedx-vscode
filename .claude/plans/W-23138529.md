# W-23138529: ESLint rule `local/no-successive-annotate-current-span`

## Context

- ESLint rule flags back-to-back `Effect.annotateCurrentSpan`, autofix → single merged call.
- `packages/eslint-local-rules` (`@salesforce/eslint-plugin-vscode-extensions`). Rule src per-file in `src/`, registered in `src/index.ts`, tested via `@typescript-eslint/rule-tester` in `test/`.
- Convention ref: `src/requireEffectFnSpanName.ts` (RuleCreator.withoutDocs, CallExpression visitor), `src/noVscodeUri.ts`. Fixer + test-output ref: `src/noEffectFnWrapper.ts`, `test/no-effect-fn-wrapper.test.ts`.
- Enabled in Effect block: `eslint.config.mjs` ~629-695 (same `files` glob as `effect/no-import-from-barrel-package`).
- Real successive usage (verified via `rg "annotateCurrentSpan"` over all non-test `*.ts`, 2026-06-23):
  - generator (consecutive `yield*`):
    - `metadata/src/commands/generateManifest.ts:52-53` (2-arg `(key,value)`, adjacent).
    - `services/src/core/metadataRetrieveService.ts:165+167` (two 1-arg ObjectExpression `yield*` separated ONLY by a blank line — blank line = whitespace, not an AST statement → IS successive).
  - `Effect.tap` chain (two adjacent `.tap` in ONE `.pipe()`, each arrow body a single annotate):
    - `services/src/vscode/workspaceService.ts:47-48` (1-arg Identifier `info => annotateCurrentSpan(info)` adjacent to 1-arg ObjectExpression `() => annotateCurrentSpan({...})` → merge `{ ...info, folders, home, workspaceName }`).
- NOT successive (valid-test cases, confirmed in tree):
  - `projectInit.ts:45,47` (generator, intervening `const dirsToCreate` between the two annotates).
  - `connectionService.ts:371` (`Effect.all`-style array `[annotateCurrentSpan(...), SubscriptionRef.set(...)]` — second element is a different effect, not an annotate).
  - `packageInstall.ts:172-185` (NOT a violation: 3 separate `.pipe()` calls, each with 1 `.tap(annotate)`; no adjacency within a pipe. Rule MUST NOT touch. Add explicit valid-test case.)
- NO `Effect.all`/`forEach` array nor `andThen` chain anywhere contains successive `annotateCurrentSpan` elements. Single-arg object form `({...})` exists abundantly standalone but never back-to-back.

### Scope (strictly successive only)

- ≥2 adjacent `Effect.annotateCurrentSpan`, no intervening statements/elements.
- Signatures merge → 1 object call:
  - `(key, value)` → `key: value` (string-literal → ident/quoted)
  - `({...})` → spread props
  - `(keyVar, val)` → `[keyVar]: val` (computed)
- Forms (scoped to verified real usage): **generator (consecutive `yield*`)** and **`Effect.tap` chain (two+ adjacent `.tap` in ONE `.pipe()`, each w/ annotate-only arrow)** ONLY. Array (`Effect.all`/`forEach`) and `andThen` detection are explicitly OUT OF SCOPE — no real usage; adding speculative AST traversal risks false positives. Add later if usage surfaces.
- Tap-chain adjacency is WITHIN a single `.pipe(...)` argument list. Separate `.pipe()` calls (even on adjacent `const` lines, e.g. packageInstall.ts) are NOT a chain — each tap is the sole annotate in its own pipe. Rule MUST NOT merge across pipe boundaries.
- Duplicate keys (same merged key appears ≥2×): report `duplicateKey` error. Resolution is part of the single autofix (keep-last); NOT a separate ESLint suggestion.
- Autofix: replace run w/ single `Effect.annotateCurrentSpan({ ...merged })`.

## Phases

### Phase 1 — rule + unit tests + register

- `src/noSuccessiveAnnotateCurrentSpan.ts`: rule `no-successive-annotate-current-span`, `type: 'suggestion'`, `fixable: 'code'`, msgs: `successiveAnnotate`, `duplicateKey`. No ESLint `suggest`/`hasSuggestions` (no existing rule uses them; dup-key resolved inside autofix). `meta.hasSuggestions` omitted.
  - `isAnnotateCurrentSpanCall` (Effect member check).
  - Arg→prop extraction: 1-arg ObjectExpression → props; 1-arg non-Object Identifier/expression → spread `...expr` (covers `annotateCurrentSpan(info)`); 2-arg literal key → prop; 2-arg non-literal key → computed `[k]`.
  - Tap-chain fixer: merged single `.tap` keeps the FIRST tap's arrow param (e.g. `info =>`) since merged props may reference it (`...info`); drop the trailing taps.
  - Group adjacency per form (two forms only):
    - generator: consecutive `ExpressionStatement` whose expr is `YieldExpression` (delegate) wrapping an annotate call; adjacency is by statement order — blank lines/comments between two annotate `yield*` do NOT break the group (no intervening AST statement). Break group only on a non-annotate statement.
    - tap chain: two+ `Effect.tap(...)` that are ADJACENT elements in the SAME `.pipe()` argument list, where each arrow body is a single annotate call; break on any non-annotate `.tap`/other operator between them. Do NOT cross `.pipe()` boundaries (each `.pipe()` is a distinct CallExpression — a lone tap in its own pipe is never a chain).
  - Dup-key detection over merged props (literal + computed-literal): same key appears ≥2×. Report `duplicateKey` at the run. The autofix keeps the LAST value for the dup key (object-literal later-wins). `successiveAnnotate` is the normal (no-dup) error.
  - Fixer (both forms): replace source span first→last annotate w/ single `Effect.annotateCurrentSpan({ ...merged })`; preserve indentation/`yield*`-vs-`.tap` wrapping of the first; idempotent.
- Register `src/index.ts` (`'no-successive-annotate-current-span': noSuccessiveAnnotateCurrentSpan`).
- `test/no-successive-annotate-current-span.test.ts` (mirror no-effect-fn-wrapper.test.ts shape; `filename` under an Effect pkg, e.g. `packages/salesforcedx-vscode-metadata/src/test.ts`):
