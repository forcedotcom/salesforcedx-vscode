# W-22973006 — Refactor getRetrievedFileUri to use SDR RetrieveResult

## Context

- `getRetrievedFileUri(result: unknown)` (testController.ts:1654) re-validates SDR-typed value at runtime
- arg = success value of `MetadataRetrieveService.retrieve(...)`; success channel = `RetrieveResult` (services/core/metadataRetrieveService.ts:146 `pollStatus()` -> `RetrieveResult`; :177 returns outcome)
- caller (testController.ts:1100) narrows out `typeof result === 'string'` cancellation branch before :1105 call -> `result` already `RetrieveResult`
- hand-rolled `MetadataRetrieveOutcomeLike` / `MetadataRetrieveFileResponse` (typeGuards.ts:13-35) duplicate SDR exports
- SDR root re-exports `RetrieveResult`, `FileResponse`: node_modules/@salesforce/source-deploy-retrieve/lib/src/index.d.ts:1
- `getFileResponses(): FileResponse[]`: client/metadataApiRetrieve.d.ts:18
- `FileResponseBase.filePath?: string`: client/types.d.ts:38
- SDR present at workspace root via services (`^12.32.5`, resolved 12.35.10); apex-testing has no SDR dep (package.json:38-41)
- `isPlainObject` still used: discoveryVfs/apexTestDiscoveryStore.ts:11,30 — KEEP
- test mock returns `{ getFileResponses: () => [] }` cast (testController.test.ts:18); `getRetrievedFileUri` module-private — retype won't break callers

## Coverage reality (advocate correction)

- ZERO e2e (Playwright/WDIO) coverage exists for `sf.apex.test.orgOnlyClass.retrieve`. Prior plan claim "retrieve-open flow covered by orgOnlyClass e2e" was FALSE.
- The user-visible flow (test explorer org-only class -> codelens "retrieve" -> SDR retrieve -> open retrieved `.cls`) is exercised ONLY by mocked unit tests (testController.test.ts mock returns `{ getFileResponses: () => [] }`).
- `getRetrievedFileUri` arg is the success value of `MetadataRetrieveService.retrieve` = SDR `RetrieveResult` produced at RUNTIME by `pollStatus()` (services/core/metadataRetrieveService.ts:146). Static typing narrows the compile-time contract; it does NOT prove the runtime object matches across SDR version bumps (root SDR `^12.32.5`, currently 12.35.10 — floating).
- Decision: strong-type the signature AND retain runtime guards as genuine defense-in-depth (not "cheap but removable"). Add the missing Playwright desktop e2e so the flow has real coverage independent of the type change. Both, per finding option (1)+(2).

## Phases

### Phase 1 — refactor with retained runtime defense (single commit)

- package.json: add `@salesforce/source-deploy-retrieve` to devDependencies (type-only, erased at build); root install
- typeGuards.ts: delete unused `MetadataRetrieveFileResponse`, `MetadataRetrieveOutcomeLike`; KEEP `isPlainObject`. Re-evaluate `isMetadataRetrieveOutcomeLike` / `isMetadataRetrieveFileResponse`: keep them (still used as runtime defense below) — only delete a guard if it has zero remaining call sites after refactor.
- testController.ts:
  - imports: add `import type { RetrieveResult, FileResponse } from '@salesforce/source-deploy-retrieve'`; drop only typeGuards symbols that become genuinely unused
  - retype `getRetrievedFileUri(result: RetrieveResult): URI | undefined` (compile-time contract via SDR types)
  - body: iterate `result.getFileResponses()` typed `FileResponse[]`; `.find` first non-empty `filePath` -> `URI.file`
  - RETAIN runtime defense-in-depth: `try`/`catch` around `getFileResponses()` and `Array.isArray`/length guard. Rationale (not removable): value originates from runtime `pollStatus()`; SDR is a floating dependency; the wrapping caller (testController.ts:1116-1120) already treats this path as best-effort (open editor is optional, refresh swallows errors). A malformed runtime shape must degrade to `undefined` (skip editor open), not throw. Express the per-item filtering with strong types: `FileResponse[]` `.find(r => typeof r.filePath === 'string' && r.filePath.length > 0)` rather than the old `isMetadataRetrieveFileResponse` hand guard — keeps defense while using SDR types.
