# W-23069609 — Migrate sf.org.logout.default to services

## Context

- Cmd `sf.org.logout.default` → `orgLogoutDefault` → `OrgLogoutDefault.run` (`packages/salesforcedx-vscode-org/src/commands/auth/orgLogout.ts:112-152`).
- `AuthRemover.removeAuth(username)` ALREADY does both internally (`node_modules/@salesforce/core/lib/org/authRemover.js` `removeAuth`):
  - `unsetConfigValues` — clears ALL global+local config keys pointing at username/aliases (target-org AND target-dev-hub).
  - `unsetAliases` — removes all aliases on disk for username.
  - BUT both read aliases via the in-memory `StateAggregator` singleton (`getAliases` → `aliases.getAll`), cached per `Global.DIR` from `init()`'s `StateAggregator.getInstance()`. If the extension cached state before an alias was added, `unsetAliases` silently misses it. Mitigated in Phase 1 by `clearInstanceAsync()` before `create()`.
- So `OrgLogoutDefault.run` manual cleanup is redundant: `removeOrgAliases`, `checkIsCurrentTargetOrg` + `doUnsetTargetOrg` (lines 122-127).
- Scratch confirm via `confirmOrThrow` — DONE (`authParamsGatherer.ts:200-204`, `ScratchOrgLogoutParamsGatherer`). No change.
- `updateConfigAndStateAggregators` (`util/orgUtil.ts:115`) refreshes extension caches (ConfigAggregator, StateAggregator, ConnectionService) — NOT redundant; `removeAuth` only touches disk. Keep; add call after `removeAuth` (currently `OrgLogoutDefault.run` omits it; `OrgLogoutSelected.run` has it at line 60).
- Scope = default only. Helpers `removeOrgAliases` / `checkIsCurrentTargetOrg` / `doUnsetTargetOrg` still used by `OrgLogoutSelected` (lines 48-57) — keep defs; remove only from `OrgLogoutDefault`.
- Gated by WI 3 (services AliasService/ConfigService present) — already merged; helpers in file prove API available.

## Phases

### Phase 1 — simplify OrgLogoutDefault to rely on removeAuth

- `OrgLogoutDefault.run`: drop `shouldUnset`/`checkIsCurrentTargetOrg`, `removeOrgAliases`, `doUnsetTargetOrg` calls.
- Keep: `AuthRemover.create()` + `removeAuth(response.data)`, telemetry catch, `return true/false`.
- STALE-CACHE GUARD (adversary):
  - `AuthRemover.init()` → `StateAggregator.getInstance()` = cached singleton keyed by `Global.DIR` (`stateAggregator.js:27-34`).
  - `removeAuth` → `unsetAliases`/`unsetConfigValues` read aliases via in-memory `aliases.getAll` (`authRemover.js` `getAliases`).
  - Prior code masked this by re-reading disk via `AliasService.getAliasesFromUsername` (`removeOrgAliases`).
  - Fix: `await StateAggregator.clearInstanceAsync()` BEFORE `AuthRemover.create()` so `init()` rebuilds from disk. (`orgUtil.ts:124` clears it only AFTER `removeAuth`.)
  - Import `StateAggregator` from `@salesforce/core`.
- Add `await updateConfigAndStateAggregators()` after `removeAuth` (refresh extension caches post-disk-change; also clears the singleton again for subsequent commands).
- `orgAliases` ctor field now unused by default path → drop `aliases` param from `OrgLogoutDefault` + `new OrgLogoutDefault(aliases)` call (line 146 → `new OrgLogoutDefault()`). `aliases` still passed to `ScratchOrgLogoutParamsGatherer` (line 145), so `resolveTargetOrg` aliases stays.
- Files: `packages/salesforcedx-vscode-org/src/commands/auth/orgLogout.ts`
- Commit: `refactor(org): sf.org.logout.default relies on removeAuth for config+alias cleanup - W-23069609`
