## Phases

### Phase 1 — CodeCoverageService (Effect.Service, Ref state)

- New `src/codecoverage/codeCoverageService.ts`: `Effect.Service<CodeCoverageService>()('CodeCoverageService', { accessors: true, dependencies: [] })`.
  - **Do NOT add `SettingsService.Default` to `dependencies[]`.** SettingsService is already provided ambiently to every service in the apex-testing runtime via the prebuilt context (`prebuiltServicesDependencies` includes `SettingsService` — services/index.ts:99 — and `allServicesLayer.ts` `Layer.succeedContext(...prebuiltServicesDependencies)` injects it). Declaring it as a hard `Default` dependency double-provisions and causes a layer conflict at runtime.
  - Access it the same way ChannelService is consumed in `handleCoverageException` today: yield it directly inside the effect — `const settings = yield* SettingsService` (accessor), import the type from `@salesforce/salesforcedx-vscode-services`. (`adversary:high` — services/index.ts:99, allServicesLayer.ts.)
- State: 2 `Ref.make<Range[]>([])` (covered/uncovered).
- Move `getCoverageData`, `applyCoverageToSource`, helpers into the service body as `Effect.fn`s (or keep pure module helpers like `getLineRange`/`getApexMemberName`, call from the service effects). `handleCoverageException` moves too (see Phase 2 for its catchTags shape).
- Effects: `applyForEditor(editor, highlightEnabled)` — compute coverage, set Refs, return ranges; `clear()` — reset Refs; `getRanges()` — read Refs.

#### 1a — Tagged errors (one per distinct failure reason)

Replace the 7 `throw new Error(...)` sites with `Schema.TaggedError` classes (each with a `message: Schema.String` field; `cause`/`filePath` where it adds context). Do NOT collapse into one generic `CoverageError` — each must be `catchTag`-able individually (SKILL.md: "Every distinct failure reason deserves its own error type"):

| Throw site (colorizer.ts) | Message key | Tagged error |
| --- | --- | --- |
| L54 no workspace folder | `colorizer_no_code_coverage_on_project` | `NoCoverageOnProjectError` |
| L62 readDirectory failed | `colorizer_no_code_coverage_on_project` | `NoCoverageOnProjectError` |
| L76 no result files | `colorizer_no_code_coverage_on_project` | `NoCoverageOnProjectError` |
| L94 no recent (non-stale) files | `colorizer_no_code_coverage_on_project` | `NoCoverageOnProjectError` |
| L131 files exist, none have coverage keys | `colorizer_no_code_coverage_in_recent_results` | `StaleResultsError` |
| L240 no coverage for current file | `colorizer_no_code_coverage_current_file` (takes path) | `NoCoverageForFileError` (add `filePath` field) |
| L36 line out of document range | `colorizer_out_of_sync_code_coverage_data` | `OutOfSyncCoverageError` |

- The four `NoCoverageOnProjectError` sites share a tag but are distinct conditions (no workspace / readDir failed / no result files / all stale). Keep them as one tag (same user-facing message + same handling) but annotate each `Effect.fail` site with a one-line comment naming the condition. The remaining three conditions get their own tags.
- Export all tagged errors from `codeCoverageService.ts` (consult ts4023-effect-errors skill for the export-surface requirement that avoids TS4023 on the service's inferred error channel).

#### 1b — Effectful loops (no imperative `for`, no silent `catch {}`)

The two imperative `for` loops driving `stat`/`readFile` must become `Effect.forEach`; silent `catch {}` becomes explicit `Effect.catchAll(() => Effect.void)` with a rationale comment (SKILL.md: "`forEach`/`for` driving effects → `Effect.forEach`"; "catchAll + swallow → must" carry a reason):

- L81-91 stat loop → `Effect.forEach(resultNames, name => statRecent(name), { concurrency: 'unbounded' })` collecting recent entries. Per-file `stat` failure → `Effect.catchAll(() => Effect.succeed(Option.none()))` + comment "skip files we can't stat". (Stat order does not matter; sort happens after.)
- L112-128 read loop → `Effect.forEach(filesToRead, name => readResult(name), { concurrency: 1 })`. **Must be `concurrency: 1` (sequential):** last-write-wins aggregation into `coverageByName` depends on chronological order (`sortByMtimeAscending`); unbounded concurrency would race the `Map.set` ordering. Per-file read/parse failure → `Effect.catchAll(() => Effect.void)` + comment "skip files we can't read or parse".
- `workspace.fs.*` calls wrap in `Effect.tryPromise` mapping to the appropriate tagged error (top-level readDirectory failure → `NoCoverageOnProjectError`).

#### 1c — Settings reads via SettingsService accessor

- Replace `settings.retrieveRestorePreviousResults()` (L102) → `yield* SettingsService.getValue<boolean>('salesforcedx-vscode-apex-testing','restore-previous-results', true)`. Note `getValue` returns `T | undefined` and can fail with `SettingsError`; treat undefined as the `true` default (it is passed as `defaultValue`) and let `SettingsError` surface (it is rare; no catch).
- Replace inline `workspace.getConfiguration().get('...disable-warnings-for-missing-coverage')` (L155-157, inside `handleCoverageException`) → `yield* SettingsService.getValue<boolean>('salesforcedx-vscode-apex-testing','disable-warnings-for-missing-coverage', false)`.

- Decoration application (`setDecorations`) stays at vscode edge (handler), not service.

### Phase 2 — Thin handler glue + runtime boundary + disposal

- Rewrite `colorizer.ts` `CodeCoverageHandler`: keep as Disposable glue only.
  - ctor: subscribe `window.onDidChangeActiveTextEditor` (the one allowed runPromise boundary).
  - callback + `toggleCoverage`: `getApexTestingRuntime().runPromise(CodeCoverageService...)`, then apply decorations to editor from returned ranges.
  - No coverage state/logic in class; ranges come from service result.
- **Fix the subscription leak (`adversary:high`).** Today `window.onDidChangeActiveTextEditor(...)` (colorizer.ts:174) returns a Disposable that is silently dropped and never added to `Disposable.from(...)` at index.ts:211.
  - (a) Store the returned Disposable on the handler (`private readonly editorChangeSub: vscode.Disposable`).
  - (b) Implement `dispose(): void` on `CodeCoverageHandler` that calls `this.editorChangeSub.dispose()` — implement `vscode.Disposable` so it can join `Disposable.from`.
  - (c) Add `colorizer` to the `vscode.Disposable.from(...)` list at index.ts:211 (alongside `apexToggleColorizerCmd`, `statusBarToggle`).
- `handleCoverageException` (now in/called from service) catches via `catchTags` over the itemized errors (NoCoverageOnProjectError, StaleResultsError, NoCoverageForFileError, OutOfSyncCoverageError) — not `catchAll`. Each maps to channel-append (when disable-warnings) or `showWarningMessage`. Shared handling per tag is fine, but the catch must enumerate tags so a new failure type forces a compile decision.
- AC check: only runPromise = editor-change/toggle callback boundary.
- commit: `refactor(apex-testing): reduce CodeCoverageHandler to disposable vscode edge glue - W-23165782`

### Phase 4 — Tests (owns colorizer.test.ts)
