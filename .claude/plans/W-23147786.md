# W-23147786 — Refactor apex-testing SfCommandlet → Effect pipelines

## Context

- Replace w/ Effect pipelines: cmd fns return `Effect<void, UserCancellationError, R>`; gather→execute via `Effect.fn` (NOT `Effect.gen` — span+tracing required for cross-codebase fns), run thru `getApexTestingRuntime()`.
- Register 7 cmds via `api.services.registerCommandWithRuntime(getRuntime())` in `index.ts` (drops raw `vscode.commands.registerCommand` for these). `catchTag('UserCancellationError')` swallows cancel; ErrorHandlerService surfaces real errors. Both already baked into `registerCommandWithRuntime` (`registerCommand.ts:59,69-70`).
- Cancellation: `UserCancellationError` + `considerUndefinedAsCancellation` from `PromptService` (`promptService.ts:80-87`). Gatherers fail w/ `UserCancellationError` instead of returning CANCEL.
- Progress: `PromptService.withCancellableProgress(title)` pipeable (`promptService.ts:201`) replaces `LibraryCommandletExecutor` `withProgress` wrapper. Success/fail notify inside pipeline via `notificationService` (`utils/notificationHelpers.ts`).
- Precheck: see Phase 2 "inline project precheck" — plain `yield* ProjectService.isSalesforceProject()`. MUST NOT call `sfProjectPreconditionChecker` (it runs `Effect.runPromise` internally — forbidden inside pipelines, `preconditionCheckers.ts:21`).
- Sites: `apexTestRun`, `apexTestSuiteAdd/Create/Run`, code-action entrypoints `apexTestClassRunCodeAction`/`apexTestMethodRunCodeAction`. Files: `commands/apexTestRun.ts`, `commands/apexTestSuite.ts`, `commands/apexTestRunCodeAction.ts`.

- `runApexTests` (`commands/apexTestRunUtils.ts:53`) already Effect, takes optional `progress?`/`token?` for apex-node cancel.
- Runtime context (`extensionProvider.ts:47-54`) provides `prebuiltServicesDependencies` → `ProjectService` / `TestService` accessors usable via plain `yield*` inside the pipeline (no manual `Layer.provide`).

### Completion-sentinel contract (BLOCKER — do not skip)

`LibraryCommandletExecutor.execute` emits two channel lines: `Starting {executionName} at {t}` and `Ended {executionName} at {t}` (`commandletHelpers.ts:88,106`). Deleting class removes only source of `Ended SFDX: …`. 6 e2e specs gate completion on `Ended SFDX: …` via `waitForOutputChannelText`:

- `apexTestSuite.headless.spec.ts:132` (`Ended SFDX: Create Apex Test Suite`), `:154` (`Ended SFDX: Add Tests to Apex Test Suite`), `:181` (`Ended SFDX: Run Apex Tests`)
- `runApexTestsCodeLens.headless.spec.ts:79,106`
- `runApexTestsCommandPalette.headless.spec.ts:92,120`
- `runApexTestsFailAndFix.headless.spec.ts:204`
- `codeCoverageColorizer.headless.spec.ts:116`

**Decision: preserve the sentinel.** Add a shared Effect helper `withExecutionLog(executionName)` (apex-testing util) that wraps a pipeline: appends `Starting {executionName} at {t}` before, `Ended {executionName} at {t}` after success/failure (use `Effect.ensuring` so the Ended line emits on both paths). Append via the `ChannelService` obtained from `getServicesApi` (`channels/index.ts:22`), not the `OUTPUT_CHANNEL` Proxy directly, to stay inside Effect. `executionName` values MUST match the existing localized strings the specs assert (`Run Apex Tests`, `Create Apex Test Suite`, `Add Tests to Apex Test Suite`) — reuse the same `nls.localize` keys the deleted executors passed to `super(...)`. No spec edits.

### DiagnosticCollection ownership (BLOCKER)

Three classes each create a `vscode.languages.createDiagnosticCollection('apex-testing-errors')` with the SAME id and none is disposed in `deactivate` (`apexTestRunCodeAction.ts:38`, `apexTestRun.ts:105`, `apexTestSuite.ts:145`; `index.ts:224` deactivate disposes nothing for these). This is a pre-existing duplicate-collection bug. **Fix as part of this WI:** create ONE module-level singleton in a util (e.g. `utils/diagnostics.ts`: `export const apexTestingDiagnostics = vscode.languages.createDiagnosticCollection('apex-testing-errors')`), push it to `context.subscriptions` in activate (or dispose in `deactivate`), and thread it through the Effect pipeline (passed/imported, not a class static). `handleDiagnostics` clears it before set, sets after run — preserving the existing clear-before/set-after contract. All three class statics removed when the classes are deleted.

### ApexTestRunCacheService → Ref (BLOCKER)

`testRunCache/apexTestRunCacheService.ts:15` is a hand-rolled mutable singleton with direct field mutation (`this.lastClassTestParam = test`) + a `setContext` side-effect. Once `resolveTestClassParam`/`resolveTestMethodParam` move into Effect pipelines, the cache MUST become Effect-managed state, not a class field mutated from inside a generator. Convert to a `Ref`-backed service:

- New `Effect.Service` (`accessors: true`) e.g. `ApexTestRunCacheService` holding two `Ref<string>` (class param, method param) built in the service `effect`.
- Methods as `Effect.fn`: `getLastClassTestParam`, `hasCachedClassTestParam`, `setCachedClassTestParam` (Ref.set + `executeCommand('setContext', 'sf:has_cached_test_class', true)`), and method equivalents.
- The `setContext` call wraps in `Effect.promise` (it's fire-and-forget, no error channel needed).
- Add the service to `buildAllServicesLayer` merge so it's in the runtime context; consumers `yield* ApexTestRunCacheService`.
- Keep `isEmpty` as a plain pure helper (no Effect).

### Cancel-token nuance

- `withCancellableProgress` interrupts the fiber on Cancel but gives no vscode `CancellationToken`. apex-node `runTestAsynchronous` needs a real token to abort the network poll (`apexTestRunUtils.ts:78,80`).
- Options: (a) bridge a `vscode.CancellationTokenSource` into the pipeline so `runApexTests` token still aborts apex-node, OR (b) accept fiber-interrupt only (network keeps polling until GC). Prefer (a) — wrap effect in `Effect.acquireRelease` of a `CancellationTokenSource`, cancel source on interrupt, pass token to `runApexTests`. → decide during impl
