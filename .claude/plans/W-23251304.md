# W-23251304: Add auto-terminate setting for orphaned Apex LS processes

## Context

- Target: `packages/salesforcedx-vscode-apex/src/languageServerOrphanHandler.ts`
- New boolean setting `salesforcedx-vscode-apex.autoTerminateOrphanedProcesses` (default `false`)
- When `true`: silently kill orphans on activation, skip notification
- When `false`: show existing toast + new "Always Auto-Terminate" button
- "Always Auto-Terminate" click -> modal confirming (1) enables setting, (2) kills now, (3) can disable later; Confirm = enable + kill; Cancel = no-op

This avoids re-implementing an existing service with bare `vscode.workspace.getConfiguration()` wrappers.

### Phase 3: Implement auto-terminate logic in orphan handler

- In `checkAndResolveOrphanedLanguageServers`:
  - Yield `api.services.SettingsService` to read the setting: `(yield* api.services.SettingsService).getValue('salesforcedx-vscode-apex', 'autoTerminateOrphanedProcesses', false)`
  - If `true`, skip prompt, kill all orphans directly, return
  - Catch `SettingsError` gracefully (treat as `false` / fall through to prompt)
- In `promptOnce`:
  - Add 3rd button `nls.localize('always_auto_terminate')`
  - Handle new button choice: show modal via `showWarningMessage` with `{ modal: true, detail: nls.localize('auto_terminate_confirm_modal') }` explaining consequences; on Confirm -> persist setting via `(yield* api.services.SettingsService).setValue('salesforcedx-vscode-apex', 'autoTerminateOrphanedProcesses', true)` + return `true` (kill); on Cancel -> return `false`
  - Catch `SettingsError` from `setValue` on the span (annotateRootSpan) so failures are observable, not silently swallowed
- Export nothing new; all setting access is inline via the Effect pipeline using the existing service

### Phase 4: Unit tests

- Update `setWarningChoices` helper and `Choices` type to add a third discriminator queue (`autoTerminateConfirm`) for the auto-terminate confirmation modal. The discriminator must key on a substring unique to the `auto_terminate_confirm_modal` message body (the modal `detail` text), because `message.includes('Terminate them?')` does NOT match the new dialog -- without this fix the new modal responses would silently fall into the main `queue`, corrupting the response sequence for both new and existing tests.
  - Updated routing: `message.includes('Terminate them?') ? confirmQueue.shift() : message.includes(<auto_terminate_confirm_unique_substring>) ? autoTerminateConfirmQueue.shift() : queue.shift()`
- Add `SettingsService` stub to `makeApi` returning a mock that tracks `getValue` / `setValue` calls
- Test: setting `true` + orphans found -> kills without prompt
- Test: setting `false` + user clicks "Always Auto-Terminate" + confirms modal -> `setValue` called with `true` + kills
- Test: setting `false` + user clicks "Always Auto-Terminate" + cancels modal -> no kill, `setValue` not called
- Test: setting `false` + existing flows unchanged (terminate, show, dismiss)
- Test: `SettingsService.setValue` failure -> span annotation recorded, kill still proceeds (graceful degradation)
