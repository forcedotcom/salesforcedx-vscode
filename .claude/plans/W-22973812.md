### What WDIO actually covers (the port contract)

1. `Run SFDX: Authorize a Dev Hub` (lines 50-58): assert status bar = "No Default Org Set"; then auth dev hub **via CLI auth file**, NOT UI (`authorizeDevHub(testSetup)`). In CI the dev hub is already globally auth'd (orgE2E.yml `--set-default-dev-hub` alias `hub`), so the Playwright spec does NOT re-auth â€” it reads the existing global dev hub alias via `sf config get target-dev-hub --json`.
2. `Run SFDX: Set a Default Org` (lines 60-125): click the "No Default Org Set" status-bar item; verify the 5 SFDX action commands appear in the dropdown (lines 78-100); select dev hub by alias; wait for `/SFDX: Set a Default Org successfully ran/`; **assert output panel "Salesforce Org Management" contains `target-org  ${devHubAlias}  true`** (lines 115-120); assert status bar shows dev hub alias.
3. `Run SFDX: Create a Default Scratch Org` (lines 127-136): calls `createDefaultScratchOrg()` â€” this helper **does drive the 3 UI prompts** (`executeQuickPick('SFDX: Create a Default Scratch Org...')` â†’ `prompt.confirm()` for the def-file picker â†’ `prompt.setText(alias)` + confirm â†’ `prompt.setText('1')` + confirm â†’ waits `/SFDX: Create a Default Scratch Org\.\.\. successfully ran/`). Confirmed in `node_modules/@salesforce/salesforcedx-vscode-test-tools/lib/src/salesforce-components/authorization.js` `createDefaultScratchOrg` (the exported `Promise<string>` variant at authorization.d.ts:27, NOT the `*ViaCli` variant). 4. `Run SFDX: Set the Scratch Org As the Default Org` (lines 138-156): WDIO `reloadWindow` then runs `SFDX: Set a Default Org` palette command, finds the (dynamic) scratch alias in the quick pick, waits for `/SFDX: Set a Default Org successfully ran/`, asserts status bar shows the scratch alias. WDIO does NOT verify "both orgs listed" â€” it only selects the scratch alias.

### Why no window reload is needed (replacing WDIO's `reloadWindow`)

WDIO reloads (line 142) before switching to the scratch org. The new spec does NOT reload, and this is justified by code: `setDefaultOrg` (`orgList.ts:187-205`) calls `AuthInfo.listAllAuthorizations()` **fresh on every status-bar click** (line 188-190) and supplements with on-disk aliases (`readAliasesByUsernameFromDisk`, line 191, comment line 194 "Supplement stale StateAggregator alias data with fresh disk data"). So each time the picker opens it re-reads auth state from disk, including the org just created out-of-process. The dropdown will therefore list the new scratch org without a reload.
- Guard against staleness: when opening the picker to switch to the scratch org, assert the scratch alias item is present **before** selecting it (Playwright auto-retries the locator); if absent the test fails fast with a clear message rather than silently. This is the only assertion added beyond WDIO, and it exists solely to make the reload-free approach robust â€” it is NOT the "verify both orgs listed" assertion the prior plan added.

## Out of scope (explicitly NOT doing)

- **Adding a stable DOM id to the org-picker status bar item (prior Phase 0).** This is a product change for testability beyond the WI's "migrate test" mandate, and it is unnecessary: WDIO already proves text/`aria-label`-based selection works for this exact element, so the port uses the same strategy. If the text selector proves flaky in practice, adding an id is a separate follow-up that requires WI-owner approval; it is not part of this migration.
- "Verify the picker lists BOTH orgs" before switching â€” WDIO does not do this (lines 138-156 only select the scratch alias). Not ported.
