### What WDIO actually covers (the port contract)

The WDIO spec has 4 `it` blocks. Mapping each to the new Playwright spec:

1. `Run SFDX: Authorize a Dev Hub` (lines 50-58): assert status bar = "No Default Org Set"; then auth dev hub **via CLI auth file**, NOT UI (`authorizeDevHub(testSetup)`). In CI the dev hub is already globally auth'd (orgE2E.yml `--set-default-dev-hub` alias `hub`), so the Playwright spec does NOT re-auth — it reads the existing global dev hub alias via `sf config get target-dev-hub --json`.
2. `Run SFDX: Set a Default Org` (lines 60-125): click the "No Default Org Set" status-bar item; verify the 5 SFDX action commands appear in the dropdown (lines 78-100); select dev hub by alias; wait for `/SFDX: Set a Default Org successfully ran/`; **assert output panel "Salesforce Org Management" contains `target-org  ${devHubAlias}  true`** (lines 115-120); assert status bar shows dev hub alias.
3. `Run SFDX: Create a Default Scratch Org` (lines 127-136): calls `createDefaultScratchOrg()` — this helper **does drive the 3 UI prompts** (`executeQuickPick('SFDX: Create a Default Scratch Org...')` → `prompt.confirm()` for the def-file picker → `prompt.setText(alias)` + confirm → `prompt.setText('1')` + confirm → waits `/SFDX: Create a Default Scratch Org\.\.\. successfully ran/`). Confirmed in `node_modules/@salesforce/salesforcedx-vscode-test-tools/lib/src/salesforce-components/authorization.js` `createDefaultScratchOrg` (the exported `Promise<string>` variant at authorization.d.ts:27, NOT the `*ViaCli` variant). The **alias is dynamic**: `TempScratchOrg_${year}_${month}_${day}_${user}_${ticks}_OrgAuth`, returned and stored in `scratchOrgAliasName` for reuse. Days = `1`, not 7.
4. `Run SFDX: Set the Scratch Org As the Default Org` (lines 138-156): WDIO `reloadWindow` then runs `SFDX: Set a Default Org` palette command, finds the (dynamic) scratch alias in the quick pick, waits for `/SFDX: Set a Default Org successfully ran/`, asserts status bar shows the scratch alias. WDIO does NOT verify "both orgs listed" — it only selects the scratch alias.

The new spec mirrors this: locate the org-picker status-bar item by its rendered label text ("No Default Org Set", then the org alias) using `STATUS_BAR_ITEM_LABEL` (`locators.ts:33` = `.statusbar-item-label`) + `hasText`. ### Why no window reload is needed (replacing WDIO's `reloadWindow`)

WDIO reloads (line 142) before switching to the scratch org. The new spec does NOT reload, and this is justified by code: `setDefaultOrg` (`orgList.ts:187-205`) calls `AuthInfo.listAllAuthorizations()` **fresh on every status-bar click** (line 188-190) and supplements with on-disk aliases (`readAliasesByUsernameFromDisk`, line 191, comment line 194 "Supplement stale StateAggregator alias data with fresh disk data"). So each time the picker opens it re-reads auth state from disk, including the org just created out-of-process. The dropdown will therefore list the new scratch org without a reload.
- Guard against staleness: when opening the picker to switch to the scratch org, assert the scratch alias item is present **before** selecting it (Playwright auto-retries the locator); if absent the test fails fast with a clear message rather than silently. This is the only assertion added beyond WDIO, and it exists solely to make the reload-free approach robust — it is NOT the "verify both orgs listed" assertion the prior plan added.

### Why a CLI `getTargetDevHub()` wrapper, NOT the `ConfigService` Effect

`ConfigService.getTargetDevHub()` (`salesforcedx-vscode-services/src/core/configService.ts:79-83`) is an Effect that runs **inside the extension host**: it `yield*`s `getConfigAggregator()`, which itself depends on `WorkspaceService` and a per-project `ConfigAggregator`. It is unreachable from the Playwright test harness — `playwright-vscode-ext` has no dependency on `salesforcedx-vscode-services` at runtime (package.json deps = `@playwright/test`, `@vscode/test-electron`, `@vscode/test-web`; `@salesforce/core` is devDependency only), runs as a plain Node process driving the `sf` CLI, and has no Effect runtime / no `ConfigAggregator` for the org workspace. The harness must read the **globally** configured dev hub (auth'd in CI by orgE2E.yml), not the workspace's config aggregator. So `getTargetDevHub()` is a thin `sf config get target-dev-hub --json` wrapper alongside the existing CLI wrappers in `orgs/` (`tryUseExistingOrg`, `extractAuthFields` in shared.ts all shell out the same way). This is the consistent idiom for this package; the Effect service is not an option.

### Existing helpers to reuse

- `selectQuickInputOptionByTyping`, `executeCommandWithCommandPalette`, `waitForNotification` (exported, index.ts).
- Output panel: `selectOutputChannel(page, channelName)` + `outputChannelContains(page, ...)` / `waitForOutputChannelText` already exist in `packages/playwright-vscode-ext/src/pages/outputChannel.ts` and are exported. NO new output helper needed.

## Out of scope (explicitly NOT doing)

- **Adding stable DOM id to org-picker status bar item.** Product change beyond WI "migrate test" scope, unnecessary: WDIO proves text/`aria-label` selection works for this element, so the port uses the same strategy. If the text selector proves flaky in practice, adding an id is a separate follow-up that requires WI-owner approval; it is not part of this migration.
- "Verify the picker lists BOTH orgs" before switching — WDIO does not do this (lines 138-156 only select the scratch alias). Not ported.
