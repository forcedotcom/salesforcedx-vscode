# W-23095743 — Retrieve/Deploy stale sourceApiVersion (#5313)

## Context

Retrieve/Deploy ignore mid-session `sourceApiVersion` edits in sfdx-project.json; user must delete `.sfdx`/`.sf` + reload.

2 caches make it stale:
- `globalSfProjectCache` (module-level, `projectService.ts:50`) caches `SfProject` 10 min. No invalidation on file change.
- `SfProject.retrieveSfProjectJson` memoizes parsed JSON on the instance (`@salesforce/core`). Reusing instance → stale JSON.

`setComponentSetProperties` (`componentSetService.ts:221`) reads `sourceApiVersion` via `project.retrieveSfProjectJson()` → gets stale value, sets `componentSet.sourceApiVersion`.

SDR `getApiVersion()` precedence (`componentSet.js:587`): `this.sourceApiVersion` → `this.apiVersion` → sfdx-project.json → org-api-version config → apex endpoint → 58.0. So the version written into the retrieved `package.xml` `<version>` reflects `componentSet.sourceApiVersion` — which is exactly the stale value we set. Fixing the cache fixes the on-disk manifest version.

Asymmetry: org-api-version (config) NOT stale — `ConfigService.getConfigAggregator` calls `agg.reload()` each desktop call. Only sfdx-project.json path stale.

Fix: subscribe to existing `FileChangePubSub`, filter sfdx-project.json, invalidate the matching `globalSfProjectCache` key. Invalidating → fresh `SfProject.resolve` → fresh instance w/ no memoized JSON. Fixes deploy + retrieve + manifest. Mirrors `configFileWatcher.ts`.

Repro artifacts (throwaway span instrumentation + spec) live on branch `worktree-repro-5313-apiversion` + `docs/repro-5313-stale-api-version.md`; NOT part of this fix. The repro's `resolvedManifestVersion` attribute (from `componentSet.getObject().Package.version`) is diagnostic-only and is NOT shipped.

## Phases

### Phase 1 — cache invalidation + watcher

Cache key parity: `projectService.getSfProject` (`projectService.ts:143-145`) keys the cache as `markerPath ?? workspacePath`, where `workspacePath` is the workspace folder fsPath (the dir containing sfdx-project.json). The watcher must invalidate the SAME key so the next `getSfProject` misses the cache.

Files:
- `packages/salesforcedx-vscode-services/src/core/projectService.ts` — export module-level `invalidateSfProjectCache = (cacheKey: string) => globalSfProjectCache.invalidate(cacheKey)` (per-key, NOT `invalidateAll`). `Cache.invalidate(key)` exists (`node_modules/effect/dist/dts/Cache.d.ts:115`). Per-key invalidation preserves unrelated workspace entries (capacity 10 = multi-root / concurrent windows); `invalidateAll` would flush sibling workspaces. Export a shared `sfProjectCacheKey(uriFsPath)` Effect (single source of truth, see below) so the watcher derives the key identically.
- new `packages/salesforcedx-vscode-services/src/core/sfProjectFileWatcher.ts` — `watchSfProjectFile` Effect.fn:
  - `yield* FileChangePubSub`
  - `Stream.fromPubSub` → `Stream.filter(e => normalize(e.uri.fsPath).endsWith(`${sep}sfdx-project.json`))` (use `node:path` `normalize`/`sep` like `configFileWatcher.ts`, not raw `.path.endsWith('/...')`, for Windows parity)
  - `Stream.debounce(Duration.millis(5))` — match `configFileWatcher.ts:39`. Invalidation is a cheap in-memory cache drop (no disk I/O); 5ms only dedupes rapid FS-watcher bursts and minimizes the stale window before a subsequent retrieve. (50ms in `watchEsrDecomposedContext` guards expensive work; not applicable here.)
  - `Stream.runForEach(event => …)`: derive `cacheKey` from the event, then `invalidateSfProjectCache(cacheKey)`.
  - cacheKey derivation MUST mirror `getSfProject`: `const markerPath = yield* readVsCodeTestWebDiskRootMarker; const cacheKey = markerPath ?? normalize(dirname(event.uri.fsPath));` — for the desktop/Node path, the workspace fsPath equals the directory containing the edited sfdx-project.json, so `dirname(event.uri.fsPath)`. For `vscode-test-web` scheme, the marker path is the cache key (the event URI fsPath is `/`); reuse the existing `readVsCodeTestWebDiskRootMarker` helper to stay identical to `getSfProject`. Implement as the exported `sfProjectCacheKey(uriFsPath: string)` Effect in `projectService.ts`, called from both `getSfProject` and the watcher.
- `packages/salesforcedx-vscode-services/src/index.ts` — import + `Effect.forkIn(watchSfProjectFile(), scope)` alongside existing watchers (~L262).

No production span/annotation changes in `setComponentSetProperties` — the fix is purely cache invalidation; the e2e asserts a durable on-disk signal (Phase 3), so no instrumentation needs shipping.

commitMessage: `fix(services): invalidate SfProject cache on sfdx-project.json change - W-23095743`

### Phase 3 — regression e2e (invert repro)

Assertion strategy: durable on-disk signal, NO span introspection. After retrieve, SDR writes a `package.xml` whose `<version>` element comes from `ComponentSet.getApiVersion()` (precedence #1 = `componentSet.sourceApiVersion`, which `setComponentSetProperties` sets from the freshly-read sfdx-project.json). Asserting the retrieved `package.xml` `<version>` proves the non-stale value flowed through. This avoids shipping throwaway instrumentation and avoids the private `getObject()`/`resolvedManifestVersion` path the repro used.

Precondition to make the assertion meaningful: ensure org-api-version config does NOT independently dictate the version. `getApiVersion` prefers `this.sourceApiVersion` over `this.apiVersion`, so a set org-api-version still loses to sourceApiVersion; but to keep the signal unambiguous, choose two distinct sourceApiVersion values (64.0 → 62.0) and assert the package.xml tracks the edited 62.0, not the warmed 64.0. (If a retrieve does not emit package.xml in the output dir, retrieve into a temp manifest dir / use the manifest-based retrieve path, or read the `<version>` from the retrieve result's `getPackageXml()` is NOT available in e2e — read the file on disk.)

Files:
- new `packages/salesforcedx-vscode-metadata/test/playwright/specs/retrieveStaleApiVersion.headless.spec.ts` — cross-platform (web + desktop) via editor (not Node fs). Steps:
  1. setup minimal org; createApexClass.
  2. generate manifest to warm `globalSfProjectCache` with sfdx-project.json `sourceApiVersion` = 64.0.
  3. edit sfdx-project.json `sourceApiVersion` 64.0 → 62.0 through editor (triggers the watcher).
  4. allow watcher debounce + invalidation to settle.
  5. regenerate manifest; read from editor.
  6. assert manifest `<version>` = 62.0 (tracks edit, not stale 64.0). Inverts the repro.

Production fix (web): the cross-platform e2e surfaced that on web `watchSfProjectFile` keyed the cache off the event URI's dirname (`\dx-project`, backslashed because the runner UA contains "Windows") while `getSfProject` keyed off `WorkspaceService.fsPath` (`/dx-project`), so invalidation never matched and mid-session edits stayed stale. Fixed by deriving the watcher's cache key from `WorkspaceService` — the same source `getSfProject` uses.

commitMessage: `test(metadata): e2e regression for sourceApiVersion edit mid-session - W-23095743`
