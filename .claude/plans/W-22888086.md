# W-22888086: e2e SOQL save query results to CSV and JSON

## Context

- Query Data View webview has `#save-csv-button` and `#save-json-button`
- Buttons post `{type: 'save_records', format: 'csv'|'json'}` to extension
- Desktop (file scheme): triggers `showSaveDialog` â€” rendered as VS Code simple dialog (quick input) because `files.simpleDialog.enable: true` in desktop fixtures
- Success: `info_file_save_success` notification ("We saved the results as: <filename>")
- Webview opens from SOQL Builder "Run Query" (soqlEditorInstance), NOT from text-editor `dataQuery` command
- Existing `soql-builder.spec.ts` already creates a builder, runs query, sees results tab â€” then closes it
- WI directive: extend `soql-run-query.spec.ts` with new test case(s)

## Design Decisions

### Extend `soql-run-query.spec.ts` (per WI directive)
The WI explicitly says "extend packages/salesforcedx-vscode-soql/test/playwright/specs/soql-run-query.spec.ts".
Add a second `test(...)` case to the existing spec file rather than creating a separate file.
This test exercises a different entry point (text-editor code lens run -> results tab -> save buttons)
which fits naturally as a follow-on test in the same spec.

### Naming convention: no infix for SOQL specs
The three existing SOQL specs (`soql-builder.spec.ts`, `soql-query-plan.spec.ts`, `soql-run-query.spec.ts`)
omit the `.desktop.`/`.headless.` infix. The metadata package uses infixes because it has both headless
and desktop variants of the same test. SOQL specs run only on desktop â€” no infix needed.

### File-existence assertion on disk (mandatory)
WI acceptance criterion: "assert the export files are written to the workspace". Notification-only is
insufficient. Use the `workspaceDir` fixture (exposed by `createDesktopTest` at line 235) and
`fs.access()` / `fs.readFile()` after each save to assert the file exists and has non-empty content.
Pattern follows `createProject.desktop.spec.ts:88-98`.

### Save dialog interaction (showSaveDialog + simpleDialog)
`showSaveDialog({ defaultUri })` with `files.simpleDialog.enable: true` renders a quick-input widget
with a file path input pre-filled with the `defaultUri` (full path including filename). This differs from
`showInputBox` (memfs path). The interaction pattern must:
1. Wait for `QUICK_INPUT_WIDGET` to become visible
2. The input is pre-filled with the default save path â€” accept it as-is (or verify it contains expected filename)
3. Click the "Save" action button (the `openLabel` equivalent for save dialogs) â€” NOT just press Enter

**Discovery requirement:** No existing spec exercises `showSaveDialog` with `simpleDialog`. During
implementation, the first step must be to run with `page.pause()` after clicking `#save-csv-button`
to observe the exact dialog DOM. Document the confirmed locators (input selector, button name/label)
in a code comment before writing the interaction. The button may be labeled "Save" or may use a
different label â€” must be confirmed empirically.

Fallback plan if simple save dialog just needs Enter (some VS Code versions auto-accept on the file
path input): try `.fill(expectedPath)` + `page.keyboard.press('Enter')` as the simpler alternative,
but only after empirical confirmation.

### Webview iframe disambiguation
When results panel opens in `ViewColumn.Two` while builder tab is alive in `ViewColumn.One`, both
have `iframe.webview.ready` selectors. The builder spec avoids this by closing the results tab first
(soql-builder.spec.ts:117-119).

For this test: click the "SOQL Query Results" tab to make it the active editor, then scope the
frameLocator to the active editor group:
```
const resultsTab = page.locator('[role="tab"]').filter({ hasText: 'SOQL Query Results' });
await resultsTab.click();
```
After clicking, the results webview iframe is the one in the active/focused editor group. Use:
```
const resultsFrame = page
  .locator('.editor-group-container.active iframe.webview.ready')
  .first()
  .contentFrame()
  .locator('#active-frame')
  .contentFrame();
```
If `.editor-group-container.active` does not reliably scope (must verify), fallback to closing
the builder tab before interacting with results (acceptable since we don't need the builder anymore).

## Phases

### Phase 1: Add save-results test to soql-run-query.spec.ts

**Commit:** `test(soql): e2e for saving query results to CSV and JSON - W-22888086`

**File:** `packages/salesforcedx-vscode-soql/test/playwright/specs/soql-run-query.spec.ts`

Add a new `test(...)` case to the existing file. The existing test uses `dataQuery` commands (code lens,
current file, selected text). The new test reuses the same SOQL file created pattern but adds the
save-results flow:

New test case steps:
1. Setup workbench (`setupMinimalOrgAndAuth`, `waitForExtensionsActivated`, `ensureSecondarySideBarHidden`)
2. Create SOQL file with `SELECT Id, Name FROM Account LIMIT 5`, save
3. Run query via "Run Query" code lens (triggers results webview in ViewColumn.Two)
4. Select REST API in quick pick
5. Wait for "SOQL Query Results" tab to appear
6. Click the results tab to ensure it is the active/focused editor
7. Access results webview frame â€” scoped to active editor group to avoid builder iframe ambiguity
8. Click `#save-csv-button` inside webview frame
9. Interact with VS Code simple save dialog:
   - Wait for `QUICK_INPUT_WIDGET` visible
   - Verify input contains expected filename pattern (e.g. ends with `.csv`)
   - Click the save/accept button (label TBD â€” discover via `page.pause()` during dev)
   - Fallback: if dialog auto-accepts, just press Enter after verifying input value
10. Assert success notification contains "We saved the results as:" via `waitForNotification`
11. Assert CSV file exists on disk: use `workspaceDir` fixture, find `.csv` file via `fs.readdir` + filter, assert non-empty content via `fs.readFile`
12. Dismiss notification
13. Click results tab again (re-focus after notification interaction)
14. Click `#save-json-button` inside webview frame
15. Interact with save dialog again for JSON (same pattern as step 9)
16. Assert success notification for JSON export
17. Assert JSON file exists on disk: find `.json` file, verify it parses as valid JSON with records
18. `validateNoCriticalErrors`

Fixture change: destructure `{ page, workspaceDir }` in the test callback to access the workspace path.

Key patterns:
- `import * as fs from 'node:fs/promises'` for file assertions (same as createProject.desktop.spec.ts)
- `import * as path from 'node:path'`
- `test.setTimeout(180_000)` â€” query + two save dialogs + file assertions
- Use `expect(async () => { await fs.access(...) }).toPass({ timeout: 30_000 })` for file polling (file write may lag notification slightly)
- Results frame scoping: `.editor-group-container.active iframe.webview.ready` or close builder first
- Save dialog: `page.locator(QUICK_INPUT_WIDGET)` + button interaction (not bare Enter)
