## Effect-idiom decisions (resolve advocate findings)

1. **FsProvider as a layer dependency, not a bare getter inside the effect.** Service body must NOT call `getApexTestingDiscoveryFsProvider()` inside `Effect.gen`/`Effect.fn` — params-vs-deps anti-pattern; defeats per-case isolation (getter returns module singleton). Instead:
   - Define `ApexTestingDiscoveryFsProviderTag` via `Context.Tag` wrapping the existing `ApexTestingDiscoveryFsProvider` instance type. Acceptable per service-patterns.md "When Context.Tag is acceptable" (vscode-owned infra injected at runtime, not business logic).
   - Provide the production layer `Layer.sync(ApexTestingDiscoveryFsProviderTag, getApexTestingDiscoveryFsProvider)` — the ONLY place the module getter is called is layer-build time.
   - Service `dependencies: [ApexTestingDiscoveryFsProviderLive]`; body does `const provider = yield* ApexTestingDiscoveryFsProviderTag`.
   - Tests provide a fresh `ApexTestingDiscoveryFsProvider` instance per case via `Layer.succeed(ApexTestingDiscoveryFsProviderTag, new ApexTestingDiscoveryFsProvider())` — no module singleton, no `providerInstance` reset hack.

2. **`DiscoveredApexClassesIndex` → `Schema.Struct`; delete hand-rolled guard (JSON VFS boundary).**
   - `const DiscoveredApexClassesIndex = Schema.Struct({ orgKey: Schema.String, updatedAt: Schema.String, classes: Schema.Array(ToolingTestClassSchema) })` (reuse the existing `ToolingTestClass`/schema from `testDiscovery/schemas`; if only a TS type exists there, use `Schema.Array(Schema.Unknown)` typed to `ToolingTestClass[]` is NOT acceptable — declare a `ToolingTestClassSchema` or fall back to a minimal struct matching the fields actually read. Confirm during impl which exists).
   - Write path: `Schema.encode`/`JSON.stringify(Schema.encodeSync(...))` to produce the persisted bytes (encode enforces shape at write).
   - Read path: `Schema.decodeUnknown(DiscoveredApexClassesIndex)(JSON.parse(...))` replaces `isDiscoveredApexClassesIndex`. DELETE `isDiscoveredApexClassesIndex`.
   - `export type DiscoveredApexClassesIndex = Schema.Schema.Type<typeof DiscoveredApexClassesIndex>`.

3. **No swallow-all on read.** `readDiscoveredClassesIndex` returns `Effect<Option<DiscoveredApexClassesIndex>, DiscoveryReadError>`:
   - FsProvider read throws `vscode.FileSystemError.FileNotFound` for a never-saved org → map ONLY that to `Option.none` (genuine "not yet discovered", expected). Detect via `error instanceof vscode.FileSystemError && error.code === 'FileNotFound'` inside `Effect.try`/`Effect.tryPromise` catch, or `Effect.catchIf`.
   - Any other read failure or a `ParseError` from `Schema.decodeUnknown` (corrupt/garbage index) → `DiscoveryReadError` tagged failure (`Schema.TaggedError`, fields: `orgKey`, `message`, optional `cause`). Surfaced, not swallowed.
   - Do NOT carry forward the current `catch { return undefined }`.

4. **No swallow at the consumer.** `clearOrg` best-effort delete: the current `clearOrg` swallows a delete-when-absent error. Map `FileNotFound` to `Effect.void` (nothing to clear is fine, with a rationale comment); surface other delete failures as a `DiscoveryClearError`. Do not blanket-catch.

5. **Logging.** Use `Effect.log` (structured) inside the service for save/clear. No `console.*` anywhere in new code.
