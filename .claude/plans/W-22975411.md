# W-22975411 — getCliId (and getDevHubId) Effect.cached rebuilt per call

## Context

- `cliTelemetry.ts:40-44` — `getCliId()` pipes `Effect.cached` inside the function body. Each call rebuilds the whole pipeline, so the cache is per-call, never shared. Comment "Cached permanently" is wrong.
- `Effect.cached` returns `Effect<Effect<A>>` — the memo wrapper is built when the *outer* effect runs. Building it fresh per `getCliId()` call defeats memoization.
- `connectionService.ts:356-360` — `getDevHubId(devHubUsername?)` has the **identical** bug: `Effect.cached` is piped inside the function body, rebuilt every call. Difference: it takes a `devHubUsername` argument, so memoization must be **per-username** (`Effect.cachedFunction`), not a single module-scope value.
- Both are called in the same `Effect.all` at `connectionService.ts:270-283` for every `getConnection` that lacks a cached `cliId`/`devHubOrgId`. Each call spawns `sf telemetry --json` / `AuthInfo.create` and emits dup `fetchCliId`/`getDevHubId` spans per session. Fixing only one leaves a symmetric defect. **Scope expanded to cover both.**
- Established repo pattern for no-arg memo: `Effect.runSync(Effect.cached(...))` once at module scope.
- Established repo pattern for arg-keyed memo: `Effect.runSync(Effect.cachedFunction(f))` (`spanTransformProcessor.ts:92`). `cachedFunction: (f: (a: A) => Effect<B,E,R>) => Effect<(a: A) => Effect<B,E,R>>` (`node_modules/effect/dist/dts/Effect.d.ts:782`) — memoizes per distinct argument using default `Equivalence`.
- `runSync` on `Effect.cached`/`cachedFunction` is safe: it builds only the memo wrapper; the inner effect is deferred until the returned effect is run.

## Phases

### Phase 1 — memoize cliId effect at module scope (`cliTelemetry.ts`)

- Hoist memo wrapper to module scope:
  - Inner effect: `const cliIdEffect = process.env.ESBUILD_PLATFORM === 'web' ? Effect.succeed(undefined) : fetchCliIdFromCli();`
  - `const cachedCliId = Effect.runSync(Effect.cached(cliIdEffect));`
  - `export const getCliId = () => cachedCliId;` (keep the fn shape so the callsite at `connectionService.ts:280` is unchanged).
- Update the comment to describe true module-scope memoization.
- files: `packages/salesforcedx-vscode-services/src/observability/cliTelemetry.ts`

### Phase 2 — memoize getDevHubId per username (`connectionService.ts`)

- `getDevHubId` is argument-keyed, so plain `Effect.cached` would either (a) collapse all usernames to one cached orgId (wrong) or (b) keep rebuilding per call. Use `Effect.cachedFunction`:
  - Define the per-username effect builder (the current body minus `.pipe(Effect.cached, ...)`):
    `const buildDevHubId = (devHubUsername?: string) => (devHubUsername ? createAuthInfoFromUsername(devHubUsername).pipe(Effect.map(authInfo => authInfo.getFields().orgId)) : Effect.succeed(undefined)).pipe(Effect.withSpan('getDevHubId', { attributes: { devHubUsername } }));`
  - Memoize once at module scope: `const getDevHubId = Effect.runSync(Effect.cachedFunction(buildDevHubId));`
  - Callsite at `connectionService.ts:279` is unchanged (`getDevHubId(devHubUsername)` still returns an `Effect`).
  - `withSpan` stays *inside* the memoized builder so the span (and the underlying `AuthInfo.create`) emits once per distinct username, not once per `getConnection` — matches the intended dedup.
  - Ordering note: drop the old trailing `Effect.flatten`; `cachedFunction` already returns `(a) => Effect<B>` so no nested-effect flatten is needed.
- files: `packages/salesforcedx-vscode-services/src/core/connectionService.ts`
- commit: `fix(services): memoize getDevHubId per username so AuthInfo.create runs once per devhub - W-22975411`
