# W-23275541: Reload Testing sidebar after Apex test suite deletion

## Context

- "Delete from Project and Org" on `.testSuite-meta.xml` triggers `MetadataDeployService.deploy` which publishes a `MetadataChangeEvent` with `metadataType: 'ApexTestSuite'` and `changeType: 'deleted'`
- `apexMetadataChangeWatcher` receives this, sets `pendingHasSuite = true`, calls `testController.incrementalUpdate(changes, true)`
- `incrementalUpdate` calls `clearAllSuiteChildren()` which only empties children of existing suite items — does NOT remove the suite item itself from the tree
- Result: deleted suite remains visible until manual refresh
- Create flow already works: `buildSuite` calls `clearAllSuiteChildren()` + `refresh()` after org API call

## Phases

### Phase 1: Fix suite refresh on metadata change (deletion path)

- Add private `refreshSuiteItems()` to `ApexTestController` that:
  - Awaits/drains `this.discoveryInProgress` (if set) before proceeding, mirroring the guard in `clearAllTestItems` (line 141-143), to prevent concurrent `populateSuiteItems` calls from racing
  - Removes `suiteParentItem` from `controller.items`
  - Clears `suiteItems` map and `suiteToClasses` map
  - Resets `suiteParentItem` to undefined
  - Calls `populateSuiteItems()` to re-query suites from org
- In `incrementalUpdate`, when `includesSuiteChange` is true:
  - Drain `this.discoveryInProgress` before `refreshSuiteItems()` — prevents concurrent mutations on suite state
  - Replace `this.clearAllSuiteChildren()` with `await this.refreshSuiteItems()`
- Keep `clearAllSuiteChildren()` public (clears children only) — still used by `apexTestSuite.ts:123` create/add flow
- Do NOT change `apexTestSuite.ts` — create/add pattern remains correct (suite exists on org, membership changed only)
