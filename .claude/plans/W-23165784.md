### Ended SFDX: sentinel — scope to run-path only

- Sentinel emitted ONLY in command files (`commands/apexTestRun.ts` etc.) via `ChannelService.appendToChannel(`Ended ${executionName}`)`. The Test-Explorer `executeTests` path does NOT emit it today (verified: `executeTests` ends with `notificationService.showSuccessfulExecution`, no `Ended`).
- e2e coverage today: `testExplorer.headless.spec.ts:153` asserts `Ended SFDX: Run Apex Tests` but ONLY inside the `re-run last method` step (command-palette → `apexTestRun.ts`). The two tree-item Run steps (class run ~L109-120, method run ~L123-138) assert Pass Rate / progress-gone, NOT the sentinel. So the Explorer `executeTests` sentinel is currently UNCOVERED by e2e.
- **Decision: emit the sentinel in the RUN path only** (matches the run-command behavior users already see). The execution service emits `Ended ${executionName}` via `ChannelService` at the end of `executeTests`'s run completion. net-new for Explorer path → add e2e assertion (Phase 5).
- **Debug path: NO sentinel.** `debugTests` (testController.ts:1397) delegates to `vscode.commands.executeCommand('sf.test.view.debugTests', …)`, registered in `salesforcedx-vscode-apex-replay-debugger/src/index.ts:212` → `setupAndDebugTests`, which never calls ChannelService. Emitting `Ended SFDX: Debug Apex Tests` after `debugTests()` returns would be net-new with non-deterministic completion ordering (replay-debugger session is async, not awaited to completion). Do NOT add a debug sentinel. No `Ended SFDX: Debug Apex Tests` spec exists or is planned.
- ChannelService is a declared `dependency` (see Phase 2), `yield*`-ed directly — not via `api.services`.

### Notifications / channel (per WI)

### Phase 2b — Extract ApexTestExecutionService

- New `src/views/apexTestExecutionService.ts`: `Effect.Service<ApexTestExecutionService>` `accessors: true`, `dependencies: [ChannelService.Default, SettingsService.Default, FsService.Default]`. These exist in `salesforcedx-vscode-services/src/vscode/` ({channel,settings,fs}Service.ts). `yield*` each directly in the service body — do NOT reach them via `api.services.X` (accessor bridge obscures the dependency edge, breaks dead-dep detection + test-layer substitution for the Phase 4 sentinel assertion).
- Move run/debug/result-processing logic: `runTests`/`debugTests`/`executeTests`/`updateTestResults`/`clearStaleTagsForTests`/`onResultFileCreate` + helpers `cacheSingleSelection`/`getTempFolder`.
- `lastProcessedResultFile` is `URI | null` on the shell today (anti-pattern). Model it as `Ref<Option<URI>>` built in the service layer (`yield* Ref.make(Option.none<URI>())`), yielded inside the service body. `executeTests` writes via `Ref.set`; `onResultFileCreate` reads via `Ref.get` + `Option.match`. Do NOT pass it as a setter callback param (params-vs-deps violation; null sentinel forbidden).
- `ExecutionContext` param type (mirror 4.1 `DiscoveryContext`) — runtime data only: `controller`, tags `orgOnlyTag`/`inWorkspaceTag`/`staleTag` (staleTag is required — `applyStaleTags` reads `this.staleTag!` at testController.ts:490/503/519 to tag method/class/suite items), `ensureInitialized`/`getConnection`/`getTestService`, `resolveSuiteChildren`, `getSuiteToClasses`. Tree maps read via 4.1 accessors (`ApexTestTreeService.getMethodItems` etc.), NOT params. NOT in ExecutionContext: ChannelService/SettingsService/FsService (deps), lastProcessedResultFile (now a Ref).
- `Effect.fn` per method (span names); `Effect.gen` only for service body.
- Errors: the `Schema.TaggedError`s from Phase 2a (`PayloadBuildError`, `SuiteNameUnresolvedError`, `TestTempFolderError`) plus `TestExecutionError` for run failures; `catchTags` not `catchAll`; no swallow.
- Notifications → `Effect.sync(() => vscode.window.show*)`. Per-test `run.errored`/`run.started` in `Effect.sync`.
- **Sentinel (run path only):** at run completion `yield* channelService.appendToChannel(`Ended ${executionName}`)`. No debug sentinel.
- Telemetry (`telemetryService.sendEventData`) preserved.
