# W-23069614 — Migrate sf.org.display.username to services

## Context
- Sibling of `sf.org.display.default` (WI 4.8). Picker path, not default-org path.
- Current: `index.ts registerCommands()` → `sf.org.display.username` → legacy `orgDisplay` (flag `--target-org`) → `SfCommandlet(sfProjectPreconditionChecker, SelectOrgForDisplay, OrgDisplayExecutor)`.
- `SelectOrgForDisplay` already Effect-based (`parameterGatherers/selectOrgForDisplay.ts`): `gather()` returns `{username}` or fails `UserCancellationError`. "picker from WI 2" = already present.
- `OrgDisplayExecutor` resolves username (picker selection wins) → `getOrgInfo(username)` → warning + `formatOrgInfoAsTable` → channel.
- Goal: replace legacy commandlet path w/ Effect command registered via `registerCommandWithLayer(AllServicesLayer)`, matching orgOpen/orgDelete.default pattern.
- `ConnectionService.getConnection()` resolves ONLY default org; no by-username variant. Picker yields arbitrary username, so we build an Effect-native by-username path (see Phase 0) rather than reuse the default-org connection service. Do NOT depend on 4.8's `getOrgInfoFromConnectionEffect` (4.8 unmerged; standalone branch).
- Gated by WI 3 (services POC `sf.org.open`, merged: `registerCommandWithLayer` available). No extra gate work.

## Existing Effect primitives to build on (no new runtime needed)
- `util/configAggregatorEffect.ts` `getConfigAggregatorEffect` — Effect yielding ConfigAggregator from `api.services.ConfigService`.
- `util/orgUtil.ts` `resolveUsernameFromAliasEffect` (alias→username via AliasService) and `readAliasesByUsernameFromDiskEffect` — Effect-native, span-named, no `runPromise`.
- These mean the entire by-username org-info path can be expressed in Effect with `ConfigService`/`AliasService` from context. There is no need for `Effect.promise`/`Effect.runPromise` anywhere in the new path.

## Anti-pattern being fixed (must not be carried forward)
- `util/orgDisplay.ts:46-48` `resolveUsername` calls `Effect.runPromise(getConfigAggregatorEffect.pipe(Effect.provide(AllServicesLayer)))` inside an async helper. This is the FORBIDDEN "runPromise inside service code" pattern (effect-best-practices: Anti-Patterns). Wrapping `getOrgInfo` in `Effect.promise` would re-bury this violation behind the new Effect command. We surface and remove it by extracting Effect-native versions.
- `getOrgInfoWithError(username, error: any)` (line 177) uses untyped `any` catch. Carried through `Effect.promise` this becomes an untyped `unknown` die in the error channel. Fixed by typed errors in the Effect path.

## Phases

### Phase 0 — Effect-native org-info (remove runPromise; typed errors)
In `util/orgDisplay.ts`:
- Define exported `Schema.TaggedError` types (effect-best-practices: explicit-over-generic):
  - `NoUsernameError { message }` — replaces `throw new Error(messages.no_username_provided)`.
  - `OrgInfoConnectionError { username, message, cause? }` — represents connection/query failure; carries the resolved username + a string `cause` derived from the caught error (no `any`).
- `resolveUsernameEffect = Effect.fn('orgDisplay.resolveUsername')(function* (username?: string) { ... })`:
  - if no `username`, read config via `yield* getConfigAggregatorEffect` (NOT `runPromise`); `getPropertyValue(TARGET_ORG)`.
  - if still none → `yield* new NoUsernameError({ message: messages.no_username_provided })`.
  - resolve alias via `yield* resolveUsernameFromAliasEffect(usernameOrAlias)`.
  - Pulls `ConfigService`/`AliasService` from context (provided by `AllServicesLayer` at the command boundary). No layer provided inside the helper.
- `getOrgInfoEffect = Effect.fn('orgDisplay.getOrgInfo')(function* (username?: string) { ... })`:
  - `resolvedUsername = yield* resolveUsernameEffect(username)`.
  - wrap the @salesforce/core connection/query work (`AuthInfo.create`/`Connection.create`/`Org.create`/SOQL) in `Effect.tryPromise({ try, catch })`; `catch` narrows the caught value to build `OrgInfoConnectionError` (string message via `getConnectionStatusFromError`), NOT `any`.
  - on connection failure, mirror current behavior: still return a populated `OrgInfo` with error/connection status (the legacy `getOrgInfoWithError` fallback), so the table still renders for offline/expired orgs. This is intentional graceful degradation, not error-swallowing: implement as `Effect.catchTag('OrgInfoConnectionError', err => buildErrorOrgInfoEffect(err.username, err))` returning `OrgInfo`, keeping the failure typed up to the catch point.
  - `getOrgInfoWithError`/`getOrgInfoFromConnection`/`getAllAliases`/`getConnectionStatus` become Effect-native (use `readAliasesByUsernameFromDiskEffect`); drop the `error: any` signature — caught values typed `unknown` and passed to `getConnectionStatusFromError` (which already accepts `err: any` internally; do not widen further).
- Keep the promise-based `getOrgInfo(username)` export ONLY if `display.default` legacy executor still references it on this branch. If so, redefine it as a thin wrapper `getOrgRuntime().runPromise(getOrgInfoEffect(username))` (runtime invoked at the legacy boundary, never inside a service/Effect) so the runPromise-in-helper anti-pattern is gone. Verify references before deleting.
- commit: `refactor(org): Effect-native getOrgInfo, typed errors, remove runPromise - W-23069614`
