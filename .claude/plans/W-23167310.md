# W-23167310 ‚ÄĒ close WI on merged only + one PR marker (live PR) in Details__c

## Context

- `auto-build-wi.js` monitor closes WI on PR state `merged` OR `closed`.
- A rebuild after an abandoned attempt APPENDED a 2nd `<strong>PR:</strong>` marker, so Details__c held 2+ markers. `extractPrUrl` returned the FIRST ‚Üí the OLDEST/abandoned URL.
- Repro W-23125840: PR 7557 (abandoned, closed) marker appended first + 7561 (live) appended later. `extractPrUrl` ‚Üí 7557. Monitor reads 7557 `closed` ‚Üí WI set `Status__c=Closed`, no resolution. Live 7561 never examined.

### Two coupled defects

1. **Wrong URL wins (root cause).** Multiple markers + first-match ‚Üí abandoned PR checked everywhere `wi.prUrl` flows (monitor `checkPrStatePrompt`, `closeMergedPrompt`, picker re-queue). Gating close on `merged` alone does NOT fix this ‚ÄĒ turns "wrongly close WI" into "wait on abandoned-closed PR forever; live PR never examined."
2. **`closed` should not close the WI.** A `closed`-not-merged PR is an abandoned attempt, never completion. Only `merged` is terminal success.

### Decision: one marker, held at write time
`<strong>PR:</strong>` is the special findable marker (vs arbitrary GitHub URLs in the body). Rather than choosing among many at read time, enforce a **single-marker invariant**: the two write sites STRIP any existing `<strong>PR:</strong>` block before appending the new one, so a rebuild's live PR supersedes the abandoned marker. `extractPrUrl` stays single-match ‚ÄĒ the sole marker is the live PR by construction. No "pick last" heuristic; no extra `gh` calls/tick.
