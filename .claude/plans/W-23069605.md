# W-23069605 — 4.2 Migrate sf.org.login.web to services

## Context

- Migrate `sf.org.login.web` command off `SfCommandlet` + `SfCommandletExecutor` (rxjs `CliCommandExecutor`) to Effect via services. Pattern = orgOpen POC (`commands/orgOpen.ts`, d4c8eac8e) + `orgDeleteDefaultCommand` (`commands/orgDelete.ts`).
- Gated by WI 3 (services layer w/ `TerminalService.simpleExec` env param, `withCancellableProgress`, `registerCommandWithLayer`) — already on branch base.
- Scope = `sf.org.login.web` ONLY. NOT `sf.org.login.web.dev.hub` (separate WI), NOT access-token.
- Current: `commands/auth/orgLoginWeb.ts` — `OrgLoginWebExecutor` streams output, `ProgressNotification.show`, manual stderr accumulation, port-conflict detection (`isAuthPortConflictError`), `showVerificationCodeIfNeeded()` fired non-blocking.
- Already Effect (reuse as-is): `AuthParamsGatherer` (`commands/auth/authParamsGatherer.ts`) returns `{alias, loginUrl}` via `runGatherer`; cancel = `UserCancellationError`.
- `registerCommandWithLayer` forwards `...args` + auto-swallows `UserCancellationError` + routes other failures to `ErrorHandlerService.handleCause`. Command sig `(instanceUrl?, reauthAliasOrUsername?)`.
- Port-conflict needs custom render (notification + "Show Output" button + kill steps) — diverges from generic `ErrorHandlerService`. Catch `TerminalServiceError` in command; map via `isAuthPortConflictError`; non-conflict failures rethrow to generic handler.
- Notification/channel pattern (uniform, NO legacy singleton): render the port-conflict notification with `vscode.window.showErrorMessage` directly — same pattern `errorHandlerService.ts:83,89` uses (`vscode.window.showErrorMessage(baseMessage, viewSuggestions)` + `channel.show()` on action). The legacy `notificationService`/`channelService` singletons (`@salesforce/salesforcedx-utils-vscode`) carry init-order coupling; orgDelete only still touches the `channelService` singleton for `showChannelOutput()` as residue. New command takes neither: get the channel via `api.services.ChannelService` and call its `showChannel` Effect (`channelService.ts:22`, `channel.show()`) for the "Show Output" action. Removes the hidden dependency on singleton initialization order the advocate flagged.
- Verification-code modal is Code Builder only (`CODE_BUILDER_STATE` env); keep forked non-blocking (`Effect.forkDaemon`), don't block login.
- `simpleExec`: auto-injects `SF_JSON_TO_STDOUT`+`FORCE_COLOR=0` for `sf ` cmds; default 30s timeout (raise — interactive web login slow); threads AbortSignal so `withCancellableProgress` Cancel kills child.

## Phases

### Phase 1 — migrate command to Effect + TerminalService

- New `orgLoginWebCommand` `Effect.fn` in `commands/auth/orgLoginWeb.ts` (replace class executor + thin wrapper).
- Precondition: `api.services.ProjectService.getSfProject()` (sets `sf:project_opened`, fails typed on no-project) — replaces `sfProjectPreconditionChecker`.
- Gather: run `gatherAuthParams`-equivalent inline OR reuse exported gatherer effect. Prefer extracting gatherer effect from `authParamsGatherer.ts` so command yields it directly (avoid `runGatherer` Promise round-trip). If extraction too broad, call `new AuthParamsGatherer(...).gather()` via `Effect.promise` — note decision in commit.
- Build cmd string: `sf org login web --alias <alias> --instance-url <loginUrl> --set-default` (+ `--json`). Quote/escape alias+url.
- `TerminalService.simpleExec({ command, parse: identity, timeout: <~few min> })` piped through `promptService.withCancellableProgress(localized title)`.
- Verification code: fork `showVerificationCodeIfNeeded` non-blocking before/around exec (`Effect.forkDaemon`), Code Builder only.
- Success: `yield* channel.appendToChannel(output)` (channel = `api.services.ChannelService`) + success message, `updateConfigAndStateAggregators()` (`Effect.promise`). No singleton channel.
- Failure: `Effect.catchTag('TerminalServiceError', ...)` → `isAuthPortConflictError(stderr/message)` → port-conflict notification via `vscode.window.showErrorMessage` (message = `org_login_web_port_conflict_notification_message` + `getPortKillInstructions()`; action = `org_login_web_show_output_button_text`); on action selection, `yield* (yield* api.services.ChannelService).showChannel`. NO `notificationService`/`channelService` singleton. else rethrow to generic handler.
- Keep `authErrorParser.ts` + `verificationCode.ts` (still used). Remove dead executor imports (`CliCommandExecutor`, `ProgressNotification`, `SfCommandlet`, etc.).
- Register: `commands/index.ts` export `orgLoginWebCommand`; `index.ts` drop `registerCommand('sf.org.login.web', orgLoginWeb)` from `registerCommands()`, add `yield* registerCommand('sf.org.login.web', orgLoginWebCommand)` in `activateEffect` (alongside orgOpen/orgDelete). Add constant if matching orgOpen style.
- commit: `refactor(org): migrate sf.org.login.web to services via TerminalService - W-23069605`

### Phase 3 — e2e (REQUIRED before PR lands; not deferred)

- Create `test/playwright/specs/orgLoginWeb.desktop.spec.ts` (template `orgOpen.desktop.spec.ts`, fixture `orgDesktopTest` from `fixtures/desktopFixtures.ts` — sfdx-project workspace, no org needed). One test per file.
- Real web OAuth can't complete headless → assert the port-conflict path, the only deterministic offline-provable signal. The conflict notification is a `showErrorMessage` (persistent, not an auto-dismissing info toast) so it is reliably observable per playwright-e2e "Reliable Assertions".

- **Port-binding (test-isolation contract).** Worker-scoped fixture extends `orgDesktopTest`:
  - `port1717` worker fixture: `await use(...)` after `net.createServer().listen(1717, '127.0.0.1')`; teardown `server.close()` in the fixture's after-`use` block (runs on pass AND failure).
  - Guard: if `listen` emits `EADDRINUSE`, `throw` with a clear message (fail fast, not a silent skip) — port 1717 already occupied means the host environment is dirty and the test cannot prove isolation. (CI runners are single-tenant per job; `playwright.config.desktop.ts` does not parallelize this package's specs across the same port — confirm `workers`/serialization before relying on the bind.)
  - Binding 1717 in the Playwright/Node test process blocks the `sf` child inside the extension-host Electron process because 1717 is an OS-level TCP port shared across processes on the host.

- **Driving the command past the gather QuickPick (deterministic, no blocking).** `gatherAuthParams` (`authParamsGatherer.ts:91`) shows the org-type QuickPick FIRST (production/sandbox/custom/project), THEN an alias input box. The codebase has NO helper to invoke a command with args via the VS Code API (no `page.evaluate(vscode.commands.executeCommand)` pattern exists; `executeCommandWithCommandPalette` is the only mechanism). So drive the real UI, not a bypass:
  1. `executeCommandWithCommandPalette(page, packageNls.org_login_web_authorize_org_text)`.
  2. Select the "production" org-type option in the QuickPick (localized via `packageNls.auth_prod_label`) — avoids the `custom` branch's URL input box; `production` resolves to `PRODUCTION_URL` so no second prompt.
  3. On the alias input box, press `Enter` with no text (empty string → `DEFAULT_ALIAS`, a valid answer per `authParamsGatherer.ts:128,144`). This completes gather and lets `simpleExec` launch the `sf` child, which then collides with the bound port.
  4. `waitForNotification(page, /<stable fragment of org_login_web_port_conflict_notification_message>/, { timeout: 120_000 })`; assert visible. Optionally `acceptNotification(...,'<org_login_web_show_output_button_text>')` then assert the Salesforce Org Management output channel shows.
- `test.setTimeout(180_000)` (interactive web-login child + raised simpleExec timeout).
