1. **Drop `sfProjectPreconditionChecker`.** Old `orgDelete` gated on an open SF project (orgDelete.ts:195); `orgDeleteDefaultCommand` does not (orgDelete.ts:153-191). Justified: the picker's `gather()` uses `getFreshAuthorizations()` (selectDeletableOrg.ts:25) + global auth/config services — none need a workspace project. The delete itself runs via `simpleExec` with `--target-org <username>` (no cwd dependency). Matches the migrated default command. This is an intentional precondition relaxation, not an oversight.
2. **Drop `telemetryService.sendException('org_delete_selected', ...)`** (orgDelete.ts:133). Justified: the Effect error path is `Effect.catchAllCause(cause => errorHandler.handleCause(cause))` (registerCommand.ts:39); `ErrorHandlerService.handleCause` appends to channel + shows a notification (errorHandlerService.ts:67-94). There is **no** `TelemetryService` in salesforcedx-vscode-services (`packages/salesforcedx-vscode-services/src` has no telemetry service; only `observability/cliTelemetry.ts`). `orgDeleteDefaultCommand` and `orgOpenCommand` already omit telemetry exceptions — the migrated commands rely on span/trace observability + the channel, not `sendException`. Dropping it matches the established migrated-command contract. Exception telemetry is intentionally dropped; failures remain visible via channel + error notification + spans.
3. **Remove `org_delete_selected` log/telemetry key** only after confirming no other reference (`grep org_delete_selected`). Currently only orgDelete.ts:133 references it.

## Per-org partial-failure contract (advocate-critical — Effect idiom)

The old executor loops orgs and **continues on a single failed delete**, appending `org_delete_failed_for_org` and returning `allSucceeded=false` (orgDelete.ts:108-130). This MUST be preserved.

`simpleExec` cannot fail-soft on its own: `childProcess.exec` is `promisify(exec)` which **rejects on non-zero exit** (childProcess.ts:29-34), and `simpleExec` wraps that in `Effect.tryPromise` → `TerminalServiceError` (terminalService.ts:50-54). A bare `yield* simpleExec(...)` in a loop short-circuits the fiber on the first failed org, aborting all remaining orgs. The "continue loop" behavior is impossible without explicit per-org failure capture.

Required design:

- Per org, run the delete `simpleExec` (wrapped in `withCancellableProgress`) through `Effect.either`, producing `Either<TerminalServiceError, string>` instead of failing the fiber.
- Iterate with `Effect.forEach(orgs, deleteOne)` where `deleteOne` returns the `Either` (or a discriminated `{ username; ok: boolean }`). All orgs are attempted regardless of individual failures.
- For each `Left`: `channelService.appendToChannel(nls.localize('org_delete_failed_for_org', username, orgType === 'scratch' ? 'scratch org' : 'sandbox'))`. For each `Right`: append the stdout.
- **Cancellation must still abort the whole loop**: `withCancellableProgress` interrupts the fiber (not a `TerminalServiceError`); `Effect.either` only catches typed failures in the error channel, so interruption propagates and stops remaining orgs (correct — user cancelled). Keep `Effect.either` scoped to the `simpleExec` call so it does not swallow interruption.
- After the loop, if **any** `Left` occurred, surface a failure so `handleCause`/notification fires for the user (e.g. accumulate failed usernames and `Effect.fail` a tagged error at the end, OR rely on per-org channel lines + a summary). Decision: append per-org failure lines during the loop, then after the loop `yield* Effect.fail(new OrgDeleteFailedError({ usernames }))` if any failed, so the error notification + span reflect partial failure. Successful orgs are already deleted (no rollback — matches old `allSucceeded=false` semantics where successes stand).
- `showChannelOutput()` after appending so the user sees per-org results.
- `updateConfigAndStateAggregators()` runs **after** the loop regardless of partial failure (matches orgDelete.ts:129, which runs it before the catch-free return) so the picker/cache reflect whatever was actually deleted.

### Phase 3 — e2e: orgDeleteUsername picker refresh

- New spec `test/playwright/specs/orgDeleteUsername.desktop.spec.ts`
- **Dedicated, non-shared org — never `MINIMAL_ORG_ALIAS`.** `MINIMAL_ORG_ALIAS = 'minimalTestOrg'` (minimalScratchOrgSetup.ts:14) is a single shared alias reused by `tryUseExistingOrg` across `orgPickers.desktop.spec.ts` (line 60) and `orgDeleteCommandVisibility.desktop.spec.ts` (line 23). `createDesktopConfig` runs `fullyParallel: true` with no workers cap (createDesktopConfig.ts:27), so specs run concurrently across files. A spec that **actually deletes** `minimalTestOrg` would yank the shared org out from under those parallel specs. `test.describe.configure({ mode: 'serial' })` only serializes within a single file — it cannot prevent cross-file contamination — so serialization is NOT the right tool here.
- **Approach (mirrors orgPicker.desktop.spec.ts:97,155):** create a dynamic dedicated scratch org alias `TempDeleteOrg_${Date.now()}_${testInfo.workerIndex}_${Math.random().toString(36).slice(2)}` via `sf org create scratch` (or a new dedicated helper in `orgs/`), set it as default. Because the alias is unique per worker, no other spec references it; deleting it is safe under `fullyParallel`. No `test.describe.configure` needed.
- Steps:
  - create dedicated scratch org (dynamic alias), wait for workbench, close welcome tabs, hide secondary sidebar, upsert auth fields to settings
  - gate on activation command present
  - run `SFDX: Delete Org...` (`packageNls.org_delete_username_text`); multi-select picker; toggle the dedicated org row; Enter; **accept** the confirm modal (do NOT cancel — this spec asserts a real delete)
  - delete completes: assert via `sf org display -o <alias> --json` failing / auth absent (CLI check), and channel shows success
  - **picker list refresh**: reopen the delete (or display) picker → dedicated org absent (asserts `updateConfigAndStateAggregators` refreshed the list without reload)
  - best-effort cleanup `catch` for the org alias in case the UI delete didn't complete (mirrors orgPicker.desktop.spec.ts:155-161), so a failed run doesn't leak an org
- Keep desktop-only.
