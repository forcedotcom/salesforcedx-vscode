### Phase 4 — Convert chai/mocha/mocha-snap test assertions to jest (sinon retained)
- Not a config-only swap
- No `mocha-compat` for jest; no chai in monorepo (`grep -rn '"chai"' packages/*/package.json` → none); no jest-chai bridge
- **sinon is RETAINED** — it works natively with jest (no bridge needed). 8 test files use sinon (`testService`, `syncTests`, `asyncTests`, `elapsedTime`, `codeCoverage`, `logs/logService`, `execute/executeService`, `utils/traceFlags`). Do NOT rewrite sinon stubs/spies to `jest.fn`/`jest.spyOn`; keep `createSandbox()`/`sandbox.stub()`/`.restore()` and `sinon` + `@types/sinon` devDeps. Only the test *runner* (mocha) and *assertion lib* (chai) change.
- Each `test/**` file needs an assertion + hook rewrite:
- Assertions: chai `expect(...).to.equal/deep.equal/be.true/throw` + `assert(...)` → jest `expect(...).toBe/toEqual/.toBe(true)/.toThrow` (e.g. `test/execute/executeService.test.ts:13`)
- **Chai-style assertion with NO chai import (hard runtime TypeError, blocks `npm run test` green):** `test/tests/asyncTests.test.ts:168` has `expect(testResult).to.be.a('object')` — jest's `expect` has no `.to` chain (no chai import in the file, no jest-chai extension), so this throws `TypeError` at runtime. Replace with `expect(testResult).toEqual(expect.any(Object))`. Sweep all converted files for any residual chai `.to.`/`.deep.`/`.be.` chains — none may survive (`grep -rnE '\.(to|deep|be)\.' test/` → empty after conversion).
- KEEP `@salesforce/core` `TestContext` — jest-compatible, used in monorepo jest suites
- Hooks: mocha `before/after` → jest `beforeAll/afterAll`; `describe/it` carry over
- **`mocha-snap` → jest snapshots (2 files):**
  - `writeResultFiles.snapshots.test.ts` (12 calls) + `writeAsyncResultsToFile.snapshots.test.ts` (4 calls): replace `import matchSnapshot from 'mocha-snap'` + every `matchSnapshot(x)` → `expect(x).toMatchSnapshot()` (`mocha-snap` not installed: `find node_modules -maxdepth 2 -name mocha-snap -type d` → empty)
  - existing `.snap` files are Jest v1 format — do not delete/`--ci`-regenerate; first run must pass as-is
  - If first run reports mismatched keys: align `describe`/`it` titles (jest derives keys from those + call order); do not blindly `-u`
- **`jest.config.js` MUST override `isolatedModules: false`:**
  - Problem: `config/jest.base.config.js:18` sets `isolatedModules: true` in ts-jest transform options — overrides `tsconfig.json`; Phase 2 fix alone does NOT cover the jest path; ts-jest strips `const enum`s → tests get `undefined` at runtime
  - Fix: add `transform: { '^.+\\.tsx?$': ['ts-jest', { isolatedModules: false }] }` to the `Object.assign` override in `jest.config.js` (mirrors `salesforcedx-vscode-org/jest.config.js:8`)
  - Verify: `grep isolatedModules packages/salesforcedx-apex/jest.config.js` shows `false`
- Add `jest.config.js` extending `config/jest.base.config.js` (with the `isolatedModules: false` transform above); add `test/tsconfig.json` if pattern requires (see `salesforcedx-utils`)
- Drop devDeps: `mocha`, `chai`, `mocha-snap`, `@types/mocha`, `@types/chai`, `nyc`. KEEP `sinon` + `@types/sinon`.
- ~31 files, mechanical but per-file — chai assertion semantics need careful 1:1 mapping; do not batch blindly
- Goal: `npm run test -w packages/salesforcedx-apex` green (run before declaring Phase 4 done — the asyncTests:168 chai chain above fails at runtime if missed).

   - Blocked: `TraceFlagService` is an `Effect.Service` on `ConnectionService` + vscode scope (`getExtensionScope`) — published lib can't import it. Reroute lands when `logService` joins the Effect/extension layer, or `TraceFlagService` gets a connection-only (vscode-free) entrypoint.
   - Until then Phase 6 `?? ''` guards are throwaway. Do NOT harden/extend `traceFlags.ts`.

2. **Hand-rolled timeout + polling in `streamingClient.ts` `subscribe()` (lines ~152-237).** Bare `new Promise` + outer `setTimeout` + two `setInterval` polling loops with manual `clearTimeout`/`clearInterval`. Effect idiom: `Effect.timeout`/`Effect.timeoutFail` for the deadline, `Schedule.spaced` + `Effect.repeat` (or `Stream.repeatEffectWithSchedule`) for polling, fiber-interruption for cancellation. Tie to the same Effect-adoption follow-on as #1.

3. **Hand-rolled retry in `executeService.ts` `executeAnonymous()` (`while (count < 2)`, lines ~31-46).** Retries on `ERROR_HTTP_500`/`INVALID_SESSION_ID`. Effect idiom: `Effect.retry(Schedule.recurs(1))` + `Effect.catchIf` on the error predicate.

These convert when the package (or the specific call sites) join the Effect/extension layer — not as part of a verbatim history-preserving import.
