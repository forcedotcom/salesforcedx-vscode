# Services Data-Only Boundary Reshape — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Reshape the `salesforcedx-vscode-services` public API so no live third-party instance (`Connection`, `SfProject`, `ComponentSet`, SDR `DeployResult`/`RetrieveResult`) ever crosses the extension boundary — replacing the "3pp getters" with services-owned, hand-authored data DTOs plus one loan facade (`ServicesOrg`), so the published types are import-free of the Salesforce SDK and `effect` *by construction*.

**Architecture:** Add owned data types (hand-authored, zero 3pp imports) in the extension; add reshaped service methods that map real 3pp objects → owned data *inside the service implementations* (the existing `getTargetOrgInfo`→`DefaultOrgInfo` model); deprecate the 3pp getters; migrate in-repo consumers; remove the getters. The whole owned-SDK-closure apparatus (generated bundle, drift gate, conformance check) is retired — it solved the wrong problem.

**Tech Stack:** TypeScript, Wireit, Effect-TS (extension internals; `Effect.Service`, `Schema`), Jest, npm workspaces.

**Spec:** `docs/superpowers/specs/2026-06-22-services-data-only-boundary-argument-and-plan.md` (Parts 1–4). Read Part 4 for the exact owned-type shapes.

## Global Constraints

- **Branch:** `phale/W-22419571-services-owned-types` (current). NOTE: a clean-branch reset is deferred pending lead review; this plan is written to run on the current branch and is branch-agnostic. Never work on `develop`.
- **No live 3pp instance on the public API surface** — not as a return, not as a parameter, not as a closure argument. Owned types only (data DTOs + the `ServicesOrg` loan facade).
- **The published plain `-types` surface must contain zero imports** from `@salesforce/core`, `jsforce`, `@jsforce/jsforce-node`, `@salesforce/source-deploy-retrieve`, `@salesforce/source-tracking`, `@salesforce/templates`, or `effect`. (After the reshape this holds by construction — the surface is owned data.)
- **Owned types are HAND-AUTHORED** (the `@types/vscode` technique) — NOT generated by any bundler. No `dts-bundle-generator`, no `api-extractor` SDK rollup.
- **3pp→owned mapping lives inside the service implementations** (extension owns its 3pp deps), exactly as `getTargetOrgInfo` maps to `DefaultOrgInfo` today.
- **Non-breaking, staged:** add reshaped methods alongside the getters, deprecate, migrate, remove. The getters keep working until Phase E.
- **Single source of truth:** owned types feed `contract.ts`; the Effect services and `PlainServicesApi` are both typed from it (existing alignment mechanism). No new gates.
- **Out of scope:** the Effect `services` sub-object internals (in-repo, effect-coupled); `effect-ext-utils` publish; rebuilding jsforce (`ServicesOrg` is curated, not the full `Connection`).
- **Copyright header** (BSD-3-Clause, `Copyright (c) 2026, salesforce.com, inc.`) on every new `.ts`.
- **Use npm scripts (wireit)** for compile/lint/test, not raw `npx tsc`. Pre-commit hook runs workspace lint/compile and cannot be bypassed.
- **`ServicesOrg` operation set** (covers 100% of audited `Connection` usage): `apiVersion`, `query`, `singleRecordQuery`, `create`, `update`, `delete`, `request`, `identity` — `{ tooling?: boolean }` flag where applicable; tooling is a flag, NOT a sub-object.

```ts
/* BSD header */
import type { Connection } from '@salesforce/core'; // real type — extension owns this dep, never leaks it
import type { ServicesOrg, OwnedQueryResult, OwnedSaveResult, OwnedHttpRequest, OwnedIdentityInfo, QueryOpts, ToolingOpt } from '../owned/servicesOrg';

/** Wraps a live Connection in the services-owned ServicesOrg facade. The Connection never escapes. */
export const makeServicesOrg = (conn: Connection): ServicesOrg => ({
  apiVersion: conn.getApiVersion(),
  query: async <T>(soql: string, opts?: QueryOpts): Promise<OwnedQueryResult<T>> => {
    const api = opts?.tooling ? conn.tooling : conn;
    const r = await api.query<T>(soql, { autoFetch: opts?.autoFetch, maxFetch: opts?.maxFetch });
    return { done: r.done, totalSize: r.totalSize, records: r.records };
  },
  singleRecordQuery: <T>(soql: string, opts?: ToolingOpt): Promise<T> =>
    conn.singleRecordQuery<T>(soql, { tooling: opts?.tooling }),
  create: async (sobjectType, record, opts?: ToolingOpt): Promise<OwnedSaveResult> => {
    const api = opts?.tooling ? conn.tooling : conn;
    const r = await api.sobject(sobjectType).create(record);
    return toSaveResult(r);
  },
  update: async (sobjectType, record, opts?: ToolingOpt): Promise<OwnedSaveResult> => {
    const api = opts?.tooling ? conn.tooling : conn;
    const r = await api.sobject(sobjectType).update(record);
    return toSaveResult(r);
  },
  delete: async (sobjectType, id, opts?: ToolingOpt): Promise<OwnedSaveResult> => {
    const api = opts?.tooling ? conn.tooling : conn;
    const r = await api.sobject(sobjectType).destroy(id);
    return toSaveResult(r);
  },
  request: <R>(req: string | OwnedHttpRequest): Promise<R> => conn.request<R>(req as never),
  identity: async (): Promise<OwnedIdentityInfo> => {
    const i = await conn.identity();
    return { userId: i.user_id, organizationId: i.organization_id, username: i.username, displayName: i.display_name };
  }
});

const toSaveResult = (r: { id?: string; success: boolean; errors?: unknown[] } | Array<unknown>): OwnedSaveResult => {
  const one = Array.isArray(r) ? (r[0] as { id?: string; success: boolean; errors?: unknown[] }) : r;
  return {
    id: one.id,
    success: one.success,
    errors: (one.errors ?? []).map((e: unknown) => {
      const err = e as { statusCode?: string; message?: string; fields?: string[] };
      return { statusCode: err.statusCode ?? 'UNKNOWN', message: err.message ?? String(e), fields: err.fields };
    })
  };
};
```
